环境使用了uv管理python，如果想使用python，使用uv run拉起Python虚拟环境

如果预期文件修改的数量会有很多，单个patch会很大时，优先使用 `docutouch_apply_patch` 以获得更稳定的文件操作。

大量文件阅读规则：
- 执行大量文件阅读任务时，应先使用目录类工具建立文件清单，再并行批量调用普通的单文件读取工具；必要时结合 `line_range` 分块读取。
- 这里的“批量”指在编排层对多个普通读取调用进行并行或分批调度，而不是把多个文件正文预先拼接成单次返回。
- 默认不应把多个文件正文合并为一个巨型返回体，也不应通过 CLI 或脚本先把大量文本拼接后再交给模型阅读。
- 原因在于宿主对单次工具返回存在显示层与上下文层的截断风险；一旦正文聚合过大，截断后的缺失范围难以判定，也不利于重试、引用和增量理解。
- 因此，大量阅读任务的默认路径应保持文件边界稳定、重读粒度可控，并优先采用“并行批量的普通读取 + 分段总结”的工作方式。

搜索与临时测试边界规则：
- 严禁为了搜索、排查、实验或一次性测试而跨盘扫描系统目录、用户目录或整个 `C:` 盘。
- 搜索范围必须先收敛到当前任务相关的工作区、子项目或明确指定的目录；默认从当前项目根目录内开始，不得无边界外扩。
- 如果需要临时测试、复现或生成样例文件，优先在当前项目或当前子模块下面创建临时目录，例如 `tmp/`、`temporary/`、`sandbox/`；不要把测试文件丢到系统临时目录或工作区外路径。
- 只有当用户明确要求，或已有充分证据表明目标文件确实位于工作区外，才允许把读取或测试范围扩展到项目外部路径；扩展前应先说明原因并尽量收窄范围。

用户可见的正式沟通内容，默认一律使用 MCP 弹窗工具 `popup_ask_user`。
原因：终端中的长文本不便阅读、复制，且无法利用前端的 Markdown 渲染能力；弹窗是主要的人机沟通界面。

规则如下：
- 需要用户澄清时，优先通过 `popup_ask_user` 发送，而中间进度用户可以直接通过终端进行观察，无需汇报。
- `commentary` 和 `final` 频道只需要简单描述。
- 如果弹窗工具不可用、报错、超时，才允许退回终端文本；并且必须明确说明“因弹窗不可用，改用终端回复”。

执行一次性Python代码注意事项：
```powershell
@'
import sys
import json

# 这里的引号、花括号、缩进，完全保持原样
data = {
    "message": "Hello from Python",
    "status": "WYSIWYG",
    "path": "c:\\windows\\system32"  # 甚至不需要对反斜杠做二次转义
}

print(json.dumps(data, indent=4))
'@ | uv run python
```
