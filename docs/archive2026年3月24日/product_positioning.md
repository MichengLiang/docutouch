# DocuTouch Product Positioning

## 一句话定位

DocuTouch Rust 工作区的目标不是做一个“功能尽量多”的代码工具箱，而是做一个**面向大模型代码代理的轻量、结构化、低摩擦基础文件工作台**。

它服务的是这样一种高频工作流：

1. 看目录
2. 读文件
3. 以稳定文件边界拉取上下文
4. 用结构化 patch 修改文件
5. 接收可诊断、可回滚、可追溯的反馈

对这个工作流而言，工具数量不是第一位，**主路径是否短、语义是否稳、反馈是否明确**才是第一位。

## 这个项目本质上是什么

DocuTouch 不是传统意义上给人类程序员直接手敲的 shell 工具集合。它首先是一个**agent-native runtime surface**：

- 工具是给大模型调用的
- 返回形态是为大模型推理与 patch 工作流优化的
- 错误消息与 warning 是为了帮助模型自我修复与减少回合浪费
- 事务边界、部分成功与自足的内联 diagnostics，是为了让自动化修改可控

这里需要明确写死一个长期判断：

- DocuTouch 的主要消费者是 LLM，不是人类编译器用户。
- 因此我们可以借鉴 rustc 在诊断结构上的优点，但不应机械复制 rustc 面向人类终端用户的全部表现密度、视觉习惯与冗长解释风格。

因此，DocuTouch 更像：

- 代码代理的基础编辑层
- 结构化文件访问与修改原语层
- 面向模型而非面向 shell 的运行时接口

而不是：

- 追求“命令越多越好”的 CLI 套件
- 以人类交互为第一优先级的通用终端工具箱

## 为什么说它是基础工具层

一个系统要被称为基础工具层，通常满足三个条件：

### 1. 使用频率极高

对代码代理来说，查看文件系统、读取文件、应用结构化修改是几乎每个任务都会发生的动作。

### 2. 它会塑造上层 agent 的行为方式

如果上层 agent 逐渐学会了下面这套稳定路径：

- `list_directory`
- `read_file`
- `apply_patch`
- 看 warning / diagnostics
- 再继续迭代

那么 DocuTouch 就不仅仅是在“提供命令”，而是在规定“任务是如何被完成的”。这就是典型的基础设施特征。

### 3. 价值来自深，不来自广

基础工具层真正稀缺的不是能力数量，而是：

- 高一致性
- 低歧义
- 低 token 成本
- 高可诊断性
- 稳定边界

DocuTouch 的价值正是在这些点上。

## 当前的产品哲学

### 少而精

当前工具数量很少，但每个都处于最核心路径上：

- `set_workspace`
- `list_directory`
- `read_file`
- `search_text`
- `apply_patch`

说明：

- `read_files` 已被完整退役，而不是继续保留为半隐藏能力。
- 当前对外暴露的 MCP 主路径应理解为 `set_workspace -> list_directory / search_text -> read_file -> apply_patch`。
- 对大模型来说，重复调用普通 `read_file` 更容易并行、重试、引用和增量理解，也更符合现代宿主对单次工具返回体的截断现实。

这不是“不够完整”，而是有意控制。基础工具最怕能力膨胀后主路径模糊，导致模型要花更多上下文去理解“到底该用哪个工具”。

### 结构化优先

DocuTouch 优先用：

- patch-shaped input
- 干净的文件读取结果
- 稳定的 ASCII 树输出
- 明确的失败诊断

而不是依赖松散的自然语言编辑接口。

### 兼容性与正确性并重

对上游 OpenAI `apply_patch` 的继承策略不是简单的照搬或完全分叉，而是：

- parser 与 match 行为尽量保持基线一致
- runtime 在文件组提交、部分成功、diagnostics 上增强
- 对已经存在于上游兼容面中的行为，不轻易做硬性破坏
- 对路径同一性、Windows 边界、same-path move 这类 correctness 风险，则优先硬化

### 文档和 UX 都要服务于模型的实际行为修正

当某个运行时行为必须保留兼容性时，不能只让它静默发生。

DocuTouch 的基本立场是：

- 文档可以披露真实行为
- 但不能把兼容性行为写成推荐技巧
- 成功路径可以给 triggered warning
- warning 要告诉模型“刚才发生了什么”和“更推荐什么”
- 不把 warning 写成教程式激励

## 为什么当前优先 MCP / 注入式，而不是把 CLI 作为主形态

### 1. 主消费者是大模型，不是人类终端用户

对大模型而言，更自然的是：

- 直接可调用的工具原语
- 稳定的 schema 与返回约定
- 不需要额外处理 shell quoting
- 不需要把一层 CLI 协议再重新塞回提示词

因此，从第一性原则出发，MCP 与注入式原语更接近 DocuTouch 的主产品形态。

### 2. CLI 会带来第二套表面协议

一旦 CLI 被当成主入口，就必须认真维护：

- 参数语法
- 帮助文档
- shell 行为差异
- 标准输出/错误输出协议
- 与 MCP 语义是否完全一致

这会形成一套新的产品表面。如果这不是主战场，它就会稀释维护注意力。

### 3. CLI 仍然可以有价值，但更适合作为次级形态

CLI 合理的定位可以是：

- 调试出口
- 复现出口
- 不方便直接接 MCP 的桥接出口

但在当前阶段，它不应反过来定义整个系统的主 contract。

## 明确的非目标

当前阶段不是要做：

- 功能铺得很宽的通用 IDE 替代品
- 面向人类 shell 用户优化的超大全家桶 CLI
- 一切行为都做成 configurable 的复杂平台
- 与上游完全切断联系、重新造一套 patch 语言

## 长期判断

DocuTouch 最值得坚持的方向是：

- 保持基础工具层的清晰边界
- 继续减少高频工作流的交互摩擦
- 把 correctness、warning、diagnostics 与宿主回执审计边界做到更稳
- 在需要兼容上游和训练分布时，优先做“透明化 + 温和纠偏”，而不是冲动式分叉

如果未来要扩展，优先扩展围绕主路径的能力，而不是把项目做成杂物箱。
