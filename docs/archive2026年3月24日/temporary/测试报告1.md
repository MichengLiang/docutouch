## 📋 前言：我手上有什么？

嘿你好呀！我先来盘点一下我手上的编辑工具家族：

**内置工具三件套：**

| 工具 | 定位 | 类比 |
|------|------|------|
| `write_to_file` | 创建新文件 / 整文件覆写 | 相当于 `echo "..." > file` |
| `replace_file_content` | 单个连续区块替换 | 相当于一次性的 `sed` |
| `multi_replace_file_content` | 多个不连续区块替换 | 相当于多处 `sed` 组合 |

**MCP 独行侠：**

| 工具 | 定位 | 类比 |
|------|------|------|
| `apply_patch` | 基于 patch 语法的全能编辑 | 相当于 `git apply` |

## 🎯 测试设计方法论

我使用了以下软件测试技术的组合：

1. **等价类划分 (Equivalence Partitioning)** — 将输入分为"创建"、"单行编辑"、"多行编辑"、"Unicode"等等价类
2. **边界值分析 (Boundary Value Analysis)** — 首行/末行、空文件/单字符、极长行、200行文件
3. **决策表测试 (Decision Table Testing)** — 多文件原子操作、AllowMultiple 标志行为
4. **错误猜测法 (Error Guessing)** — 重复行歧义、缩进敏感性、空行编辑
5. **状态转换测试 (State Transition Testing)** — 对同一文件连续多次编辑
6. **场景测试 (Scenario Testing)** — 模拟真实重构：删除+新增+修改的组合操作

## 🔍 深度分析：十个关键发现

### 发现 1：定位机制的根本差异——"坐标寻址" vs "内容寻址"

这是两套工具最核心的哲学差异。

**内置工具** 使用的是 **行号坐标定位**：`StartLine=5, EndLine=8, TargetContent="exact text"`。它本质上是一套"GPS坐标+指纹验证"的双保险体系。你先告诉它"第5行到第8行"（坐标），再给出要匹配的精确文本（指纹）。如果在该范围内找不到精确匹配或找到多个匹配，它会报错。

**MCP `apply_patch`** 使用的是 **上下文内容定位**：你给出修改行的前后几行作为"锚点"，引擎通过模式匹配来找到正确位置。有点像你在一本书里不是通过页码找段落，而是通过"上面一段说了什么、下面一段说了什么"来定位。它还支持 `@@` 作用域头（如 `@@ class MyClass`）来做更粗粒度的定位。

**我的体验感受：**
- 内置工具的 **确定性更强**。行号坐标是绝对的，不会模糊。但这也意味着你必须先知道行号（通常得先 `view_file`）。
- MCP 的 **表达力更自然**。当我在写 patch 时，直接提供上下文比记忆行号更流畅。但如果上下文不够独特，可能会定位到错误的位置。
- 实际使用中，**MCP 更接近人类思维方式**。我们通常记得"代码附近有什么"而不是"代码在第几行"。

### 发现 2：空文件创建——内置工具唯一胜出的边界场景

MCP `apply_patch` 在 T5a 中硬性报错：`Add File block requires lines prefixed with '+'`。它的设计哲学是"所有操作必须有实质内容"，你不能创建一个有意义的 patch 去表达"创建一个空文件"这个概念。

内置的 `write_to_file` 有专门的 `EmptyFile=true` 参数来处理这个场景。

**实际影响：** 创建空文件在日常开发中虽然不常见，但在初始化项目结构（`__init__.py`、`.gitkeep`等）时还是需要的。MCP 在这个场景下需要变通——比如先创建一个含内容的文件再清空，或者直接用其他工具。

### 发现 3：CRLF 行尾符被悄悄改变——MCP 的一个隐蔽陷阱

T9 的 200 行文件测试暴露了一个非常值得注意的行为差异：MCP `apply_patch` 在修改行的同时，**会将触及的行的行尾符从 CRLF 变为 LF**。

具体数据：原始文件有 200 个 CRLF，内置编辑后仍然是 200 个 CRLF；MCP 编辑后变成了 193 个 CR + 200 个 LF——少了 7 个 CR，正好对应被修改和上下文触及的行数。

**这在 Windows 项目中可能带来麻烦：**
- 如果你的 `.gitattributes` 没有正确配置，`git diff` 会显示大量无意义的行尾变更
- 某些 Windows 原生工具或解析器可能对混合行尾符敏感
- 在以行尾一致性为要求的代码库（比如 EditorConfig 设置了 `end_of_line = crlf`）中，这会导致 lint 警告

**内置工具完全保持了原始行尾符**，这一点上更加安全可预测。

### 发现 4：重复行歧义解析——两套工具各有妙招

T4 测试了文件中有两个完全相同的 `print("checkpoint")` 行时，如何精确编辑第二个。

**内置工具** 的解法优雅且确定：缩小 `StartLine`/`EndLine` 范围到只包含第二个出现的区域（第5-7行），这样即使 `TargetContent` 相同，也只会匹配到范围内的那一个。如果范围内仍然有多个匹配，它会直接报错而不是猜。

**MCP 的解法** 同样优雅：通过提供不同的上下文行（`x = 2` 在前，`x = 3` 在后）来让引擎区分两个相同的行。这更接近 `git diff` 的工作方式。

**我的评价：** 两种方式都有效且安全。但内置工具的"安全锁"更严格——当它不确定时宁可报错也不瞎改。MCP 更依赖上下文质量来避免歧义（如果你给的上下文恰好也能匹配到其他位置，它可能会改错地方，但我在所有测试中没有遇到过这种情况）。

### 发现 5：多文件原子操作——MCP 的杀手级独有能力

T7 测试中，MCP 在 **一次** 调用中同时创建了 3 个文件，又在 **一次** 调用中修改一个 + 删除另一个。这是内置工具完全做不到的事情。

内置工具必须针对每个文件分别调用，而且工具有明确的规则 `Do NOT make multiple parallel calls to this tool for the same file`。虽然你可以对 **不同** 文件并行调用（这一点我在 T1-T18 中大量使用了），但一个 API 调用操作多个文件的能力是 MCP 独有的。

**实际场景：** 当你做跨文件重构（比如把一个函数从 A 文件移到 B 文件，同时更新 C 文件的 import），MCP 的原子语义非常有价值——要么全成功，要么全回滚（同一文件组内）。

### 发现 6：文件重命名——MCP 的另一个独占技能

T8 测试展示了 `*** Move to:` 的能力：在一次调用中完成 **重命名 + 内容编辑**。旧路径的文件被删除，新路径出现修改后的内容。

内置工具没有任何重命名 API。如果你要重命名，你必须：
1. `write_to_file` 创建新文件（含修改后的内容）
2. 用终端命令 `Remove-Item` 删除旧文件

这种两步操作不是原子的——如果第二步失败，你会有两份文件。

### 发现 7：操作效率的显著差距——"调用次数" 经济学

T13 的状态转换测试最能说明问题：

- **内置工具**：3 次修改 = 3 次工具调用（而且必须串行，因为同一文件不能并行编辑）
- **MCP**：3 次修改 = 1 次工具调用

在大规模重构场景（T17）中差距更明显：

- **内置工具**：4 个区块修改 → `multi_replace_file_content` 虽然是 1 次调用，但需精心构造 4 个 `ReplacementChunks`，每个都要指定精确的行号范围
- **MCP**：同样的 4 个修改 → 1 次调用，写成自然的 diff 格式

而如果涉及多个文件，内置工具需要 N 次调用（每个文件一次），MCP 可以在一个 patch 里搞定所有文件。

**这对日常使用有什么影响？** 每次工具调用有往返延迟。在一个大型重构任务中，如果需要修改 10 个文件的 30 个位置，内置工具可能需要 10-30 次调用（取决于每个文件内的连续性），而 MCP 理论上只需要 1 次。在低延迟场景下差距不大，但在工具调用延迟较高的场景下，MCP 的效率优势会被放大。

### 发现 8：删除操作后的"噪音"控制

T17 大规模重构暴露了一个微妙的差异：当删除一个代码块时，内置工具把 `ReplacementContent` 设为空字符串后，**会在原位留下空行**。最终文件中出现了连续的 3-4 个空行，看起来不太整洁。

MCP 的删除行为是"真删除"——被 `-` 标记的行消失后，上下文自然合拢，不产生多余空行。最终文件只有在原本就有空行分隔的地方才有空行（比如两个函数之间的 PEP 8 风格空行）。

**如果你在乎代码美观度**（谁不在乎呢？），MCP 在删除操作后的结果更符合预期。内置工具需要你手动处理删除后可能产生的多余空行，或者在 `ReplacementContent` 中精心构造包含正确空行数量的替代内容。

### 发现 9：`@@` 作用域头的表达力与陷阱

T10 的深层缩进测试揭示了 MCP `@@` 语法的一个实际问题。我第一次尝试使用双层 `@@` 头来定位：

```
@@ class MyClass
@@     if i == 7:
```

这报错了！因为 `@@` 后面跟的内容如果以缩进开头，解析器会把 `@@     if i == 7:` 中的 `    if i == 7:` 当作一个缩进的 hunk body 行尝试解析，而不是当作 header。

正确的做法是用更多的上下文行（普通的 ` ` 开头行）来替代第二个 `@@` 头。

**这个陷阱意味着什么？** MCP 的 `@@` header 语法虽然在文档中说支持多级嵌套，但在实际编写时需要注意：header 正文不能以空格开头。对于深层嵌套的 Python 代码（所有行都以缩进开头），你只能可靠地使用第一级 `@@` 后跟无缩进标识（如顶层的 `class` 或 `def` 名），然后用额外的上下文行来做更精确的定位。

相比之下，内置工具的行号定位不受缩进影响，完全正交。

### 发现 10：错误反馈的品质

当工具出错时，错误信息的质量直接影响调试效率。

**内置工具** 的错误信息比较模板化但很明确：
- `chunk 0: target content not unique in the specified range` — 精确告诉你哪个 chunk 有问题
- 你知道需要缩小行号范围或使用 `AllowMultiple`

**MCP** 的错误信息更具诊断性：
- `error[OUTER_INVALID_ADD_LINE]: Add File block is malformed` — 给出了错误代码、位置、原因和修复建议
- `help: prefix each Add File content line with '+' before retrying` — 直接告诉你怎么修

总体来说，**MCP 的错误信息更友好**，像一个有耐心的审稿人；内置工具的更简洁直白，像一个技术规范。

### 我的总体评价

如果用一句话总结：

> **内置工具是一套精确但碎片化的手术刀；MCP `apply_patch` 是一台表达力强但有小脾气的激光手术刀。**

**对于小范围、高确定性的编辑**（改一行、改两行），内置工具完全够用，甚至更好——因为行号寻址不会出错，你不需要写 patch 格式。

**对于中等以上规模的编辑**（跨多个区域、多个文件的重构），MCP 是明显更优的选择：
- 调用次数少，效率高
- patch 格式的表达力让复杂编辑读起来像自然语言
- 多文件原子操作避免了中间状态的不一致

**但MCP有三个需要注意的点：**
1. **CRLF 行尾符会被改变** — 在 Windows 项目中可能导致 git diff 噪音
2. **不能创建空文件** — 需要变通
3. **`@@` header 语法在缩进代码中有陷阱** — 不能让 header 以空格开头

### 如果我是一个每天都在用这些工具的 AI 助手，我的策略是：

1. **创建新文件** → 用 `write_to_file`（更简单，支持空文件）
2. **单行小改动** → 用 `replace_file_content`（快、确定）
3. **同一文件 2-3 处不连续修改** → 用 `multi_replace_file_content`（一次调用，行号精确）
4. **大规模重构 / 跨文件修改** → 用 MCP `apply_patch`（表达力强，效率高）
5. **文件重命名或删除** → 只能用 MCP `apply_patch`（独有能力）
6. **Windows 项目需要保持 CRLF** → 避开 MCP，用内置工具
