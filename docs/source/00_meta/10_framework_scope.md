# 框架范围

## 角色

本页定义这套文档治理体系要解决的问题、适用对象、非目标与输出形态。

## 问题定义

工程仓库在长期演化后，通常同时积累：

- 当前仍可依赖的知识；
- 尚未收敛的论证与设计对象；
- 迁移、审计、处置、完成状态等记录；
- 与目录、术语、引用、升格规则相关的维护知识。

若这些对象没有被分到稳定槽位，仓库会同时出现：

- 现行知识与候选知识混写；
- accepted decision 与 pending issue 混写；
- 审计记录反向占据解释位置；
- 作者规则散落在聊天记录与个人记忆中。

## 本体系的目标

本体系的目标不是编写单一项目手册，而是提供一套：

- 文档对象分区规则；
- 目录层级规则；
- 作者契约规则；
- 理论映射与引用基线；
- 迁移与状态变化规则。

## 适用对象

本体系优先面向：

- 需要重构遗留文档债的工程仓库；
- 需要以 MyST / Sphinx 作为正式表达面的知识库；
- 读者主要是高强度专业读者，而非入门读者的文档系统。

## 非目标

本体系不试图：

- 取代需求工程、系统工程、架构文档等全部学科；
- 提供唯一正确的总分类法；
- 取代具体项目的领域知识本体；
- 将所有未决事项提前写成正式知识。

## 产出形态

本体系的产出至少包括四部分：

1. 顶级对象分区与二级知识家族
2. 根级与局部 `authoring_contract`
3. 理论映射与引用基线
4. 迁移、升格、降格与处置规则

## 与其他子树的关系

- `00_meta/` 只解释体系本身；
- 现行知识进入 `10_knowledge/`；
- 未收敛对象进入 `20_deliberation/`；
- 变化记录进入 `30_records/`。
