# structural_search 概要设计

## 1. 对象定义

`structural_search` 是 DocuTouch MCP 的 AST 结构查询工具。它在 DocuTouch workspace 或显式 `path` 范围内运行 ast-grep 查询，并把匹配集合呈现为面向 agent 阅读的 pretty text。

该工具只承担查询职责。它不提供替换、修复、自动编辑、批量改写或代码修改入口；传入的 ast-grep 配置如果包含 fix、rewrite、replacement、transform 等改写意图，工具必须返回参数错误，并且不执行查询。

`structural_search` 的稳定输入对象是查询范围、语言、pattern、rule、引用和显示限制。它的稳定输出对象是查询头、匹配统计、结果组、证据行、capture 摘要、省略说明和下一步查询引用。

## 2. 使用者与工作流位置

`structural_search` 的直接使用者是通过 MCP 调用 DocuTouch 工具的 agent。该 agent 的常规工作流是：

```text
任务进入
  -> 使用 list_directory 建立范围
  -> 使用 search_text 获取文本候选
  -> 使用 structural_search 获取 AST 结构候选
  -> 使用 read_file 精读关键局部
  -> 继续分析或测试
```

`structural_search` 位于 `search_text` 与 `read_file` 之间。`search_text` 负责文本召回；`structural_search` 负责把文本候选压缩成结构候选；`read_file` 负责最终源码阅读。

工具输出必须支持 ReAct 式循环中的下一步动作选择。每次可注册结果都分配连接内短引用 `qN`，结果组使用 `[N]` 编号，派生查询通过 `ref` 引用历史结果。引用只在当前 MCP connection 内有效。

## 3. 设计思想

### 3.1 坐标系转换

`rg` 的坐标系是文本行。它回答“某段文本在哪里出现”。ast-grep 的坐标系是语法节点。它回答“某个语法形状在哪里出现”。`structural_search` 的主要价值是把 agent 的观察面从文本行坐标推进到 AST 节点坐标。

这个坐标系转换带来三类收益：

- 区分同名文本在不同语法角色中的出现，例如定义、调用、属性、宏调用、match arm。
- 使用 metavariable 捕获结构内部差异，例如调用参数、match arm 字段、函数签名片段。
- 使用结构关系约束结果范围，例如某个节点位于某类 enclosing item 内，或某个节点拥有特定子节点。

### 3.2 pretty text 优先

MCP 的底层参数是结构化输入，但默认结果面必须是 pretty text。默认输出不得是 ast-grep 原始 JSON。pretty text 的固定责任是降低 agent 阅读成本，并保留后续工具调用所需坐标。

默认结果必须包含：

```text
structural_search[mode] qN
query/pattern/rule summary
language
scope/path
match statistics

[1] result group title
  path:line   evidence snippet
  context/captures summary

omitted:
- ...

next:
- ...
```

### 3.3 查询完整性

本规格覆盖 ast-grep 查询能力在 DocuTouch MCP 中的完整适配。完整适配指查询表面的完整，不包含改写能力。

查询表面包括：

- 简单 pattern 查询。
- rule object 查询。
- atomic rule：`pattern`、`kind`、`regex`、`nthChild`、`range`。
- relational rule：`inside`、`has`、`precedes`、`follows`。
- composite rule：`all`、`any`、`not`、`matches`。
- metavariable capture 与 capture shape 摘要。
- rule test 与 AST explanation。
- result expansion 与 local structural context。
- 大结果省略、错误状态和解析缺口。

## 4. 与 DocuTouch 现有架构的关系

`docutouch-core` 当前承担文件系统、读取、搜索和 patch/splice/rewrite 呈现原语。`docutouch-server` 当前承担 MCP schema、workspace/path 解析、工具注册和服务分发。

`structural_search` 应沿用这一分工：

```text
docutouch-core
  负责 structural_search 的查询模型、后端调用、结果解析、分组和 pretty text 渲染。

docutouch-server
  负责 MCP 参数 schema、workspace/path 解析、工具注册、调用 core 执行。
```

工具必须复用 DocuTouch 当前 workspace 规则。相对路径按 workspace 解析；绝对路径按现有安全与路径规则解析；缺少 workspace 且没有可解析 path 时返回 workspace-required。

## 5. 不变量

`structural_search` 的概要层不变量如下：

1. 工具只执行查询。
2. 默认输出是 pretty text。
3. 证据行包含 `path:line`。
4. 大结果必须显示 `omitted`。
5. 可注册结果使用 `qN`。
6. 结果组使用连续 `[N]`。
7. 派生输出显示来源，例如 `from: q1.[2]`。
8. 参数错误、workspace-required、invalid-ref 不分配 qN。
9. parse-partial 结果必须显示缺口。
10. next 建议必须指向合法后续查询或明确说明需要重新运行查询。
