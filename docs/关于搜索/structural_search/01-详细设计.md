# structural_search 详细设计

## 1. MCP 工具面

工具名：

```text
structural_search
```

参数：

```text
structural_search(
  mode,
  pattern?,
  rule?,
  query?,
  ref?,
  path?,
  language?,
  include_tests?,
  context?,
  limit?,
  view?
)
```

参数对象：

```text
mode
  必填。查询类型。

pattern
  可选。ast-grep pattern 文本。用于简单结构匹配。

rule
  可选。ast-grep rule object。用于 atomic、relational、composite 查询。

query
  可选。面向输出的查询说明。它不参与匹配，除非 mode 明确允许把 query 解释为 pattern。

ref
  可选。结果引用。格式为 `N` 或 `qN.N`。

path
  可选。查询范围。可为文件、目录或 path 数组；解析规则与 DocuTouch 现有 path 规则一致。

language
  可选。ast-grep 语言。显式传入时优先于路径推断。

include_tests
  可选。是否包含测试文件。默认值由实现固定，输出必须显示 tests included 或 tests excluded。

context
  可选。请求返回的上下文集合。

limit
  可选。结果组显示上限。它限制默认展示，不改变 known match 统计。

view
  可选。`preview` 或 `full`。默认 `preview`。
```

`pattern` 与 `rule` 的关系：

```text
mode=find 或 mode=rule_test 时，pattern 与 rule 至少提供一个。
如果二者同时提供，rule 优先，pattern 只作为 query summary 的辅助文本；实现也可以返回 parameter-error 要求二选一。
推荐 public contract 采用二选一，降低解释歧义。
```

## 2. mode 定义

### 2.1 `find`

`find` 在选定文件集合中运行 pattern 或 rule 匹配，返回分组后的匹配集合。

必需输入：

```text
mode=find
pattern 或 rule
path 或已设置 workspace
language 或可从 path 推断 language
```

输出对象：

```text
query header
language
scope/path
pattern/rule summary
match statistics
result groups
evidence lines
capture summary
omitted
next
```

### 2.2 `expand`

`expand` 展开已有结果组。它读取连接内 registry 中的 qN/[N]，返回该组内更多 match、capture 和 enclosing 摘要。

必需输入：

```text
mode=expand
ref=N 或 ref=qN.N
```

输出必须包含来源：

```text
from: q1.[2] <group title>
```

### 2.3 `around`

`around` 返回某个 match 的局部结构上下文。目标可以来自 `ref`，也可以来自 `path:line` 形式的 `query`。

上下文对象：

```text
enclosing
  function、impl、module、class、test item 等 enclosing item。

node
  当前匹配节点的 kind、range、短文本。

siblings
  前后相邻的语法节点摘要。

children
  直接子节点摘要。

captures
  该 match 的 capture 展开。
```

### 2.4 `explain_ast`

`explain_ast` 返回某个文件局部或结果引用对应节点的 AST 局部形状。它服务 pattern/rule 编写和调试。

目标选择：

```text
ref=qN.N
query=path:line
path + query=line number
```

输出对象：

```text
source
language
node kind
local tree
candidate pattern hints
next
```

candidate pattern hints 是工具根据 AST 局部生成的辅助建议；它们不是匹配事实。

### 2.5 `rule_test`

`rule_test` 在小范围内验证 pattern 或 rule。小范围可以是单文件、单个 ref、或 path:line 周围局部。

输出对象：

```text
pattern/rule summary
test source
status: matched | no-matches | invalid-pattern | parse-partial
matches
captures
next
```

`rule_test` 不写入任何规则文件，不注册全局规则，不改变 workspace。

## 3. ast-grep 查询对象支持

### 3.1 atomic rule

`structural_search` 必须支持 ast-grep 查询中的 atomic rule 对象：

```text
pattern
kind
regex
nthChild
range
```

每个 atomic rule 的 pretty text 输出必须显示其公共摘要。长 rule 使用 compact summary。

### 3.2 relational rule

`structural_search` 必须支持 relational rule：

```text
inside
has
precedes
follows
```

relational rule 的输出必须说明关系约束命中的结构事实。例如：

```text
context: call_expression inside function_item
relation: function_item has request_approval($$$ARGS)
```

### 3.3 composite rule

`structural_search` 必须支持 composite rule：

```text
all
any
not
matches
```

composite rule 的输出必须保留 rule summary。对于 `not`，输出只说明参与匹配的正向证据；不展示被排除节点集合，除非 `view=full` 且结果预算允许。

### 3.4 constraints 与 utility rules

如果 ast-grep rule 使用 metavariable constraints 或 utility rules，工具必须把它们作为查询定义的一部分执行，并在输出摘要中显示：

```text
constraints: $ARG restricted by <summary>
utils: <names>
```

输出不需要展开完整 utility rule，除非 `mode=explain_ast` 或 `view=full` 请求规则解释。

### 3.5 改写字段

查询工具不接受改写语义。以下字段或同义对象不属于 public contract：

```text
fix
rewrite
replacement
transform-as-edit
apply
autofix
```

如果输入 rule 包含这些字段，工具返回：

```text
structural_search[find]
status: unsupported-rule-field

Problem
  Query-only structural_search does not accept edit fields in rule input.

next:
- remove edit fields and rerun the query
```

## 4. 语言解析

`language` 显式传入时，工具按该语言解析全部候选文件。语言缺失时，工具可以从 path 推断。推断规则必须可测试：

```text
单文件 path -> 根据扩展名推断。
单语言目录 -> 根据候选文件扩展名推断。
多语言目录 -> ambiguous-language。
无法识别扩展名 -> unsupported-language。
```

ambiguous-language 输出：

```text
structural_search[find]
status: ambiguous-language
scope: packages

Language could not be selected from the current scope.

candidates:
[1] typescript: 184 files
[2] rust: 73 files
[3] python: 19 files

next:
- rerun with language=typescript
- rerun with a narrower path
```

## 5. 文件选择

文件集合由 `path`、workspace、language、include_tests 和 ignore 规则共同确定。

规则：

```text
path 为空 -> 使用当前 DocuTouch workspace。
path 非空 -> 使用 path 限定范围。
relative path -> 按 workspace 解析。
absolute path -> 按现有 DocuTouch path 规则解析。
include_tests=false -> 排除测试文件与测试目录。
include_tests=true -> 包含测试文件。
```

测试文件判定应集中实现。初始判定可包含：

```text
文件名包含 test/tests/spec/specs。
路径段包含 tests、test、__tests__、spec。
Rust 文件中 *_tests.rs。
TypeScript/JavaScript 中 *.test.*、*.spec.*。
Python 中 test_*.py、*_test.py。
```

输出 scope 行必须显示 tests 选择：

```text
scope: core; tests excluded
scope: core; tests included
```

## 6. 分组规则

`find` 默认分组必须稳定。推荐分组优先级：

```text
1. path family：production / tests / fixtures / generated。
2. enclosing item：function、impl、module、class、test item。
3. node kind。
4. capture shape。
5. file path。
```

分组标题应描述当前组的主要结构共同点：

```text
[1] unit test call sites
[2] match arms in policy conversion
[3] macro invocations under command handlers
[4] capture shape: $ARG is string literal
```

当无法生成语义化标题时，使用文件或 node kind 标题：

```text
[1] call_expression matches in core/src/exec_policy.rs
```

## 7. capture 摘要

capture 输出必须短。每个 result group 默认只展示最重要的 capture shape，不展开全部捕获文本。

格式：

```text
captures:
  $CMD = shell | command_program | literal string
  $$$ARGS has 3 observed shapes
```

`expand` 可展示每个 match 的具体 capture：

```text
captures:
  $CMD = command_program
  $$$ARGS = ctx, command, approval_mode
```

长 capture 使用横向裁切：

```text
$BODY = { let command = ... [124 chars omitted] }
```

## 8. 输出状态

### 8.1 success

```text
structural_search[find] q1
query: calls to evaluate_exec_policy
pattern: evaluate_exec_policy($$$ARGS)
language: rust
scope: core; tests included
matches: 4 groups, 12 displayed matches, 31 known matches

[1] unit test call sites
  core/src/exec_policy_tests.rs:205      evaluate_exec_policy(...)
  context: function_item under test module
  captures: $$$ARGS has 2 observed shapes

omitted:
- 3 groups not shown
- 19 matches not shown

next:
- expand 1       show unit test call sites
- around 1       show local structure for first group
```

### 8.2 no-matches

```text
structural_search[find] q2
pattern: unsafe { $$$BODY }
language: rust
scope: core/src/session; tests excluded
status: no-matches

No structural matches found in selected scope.

next:
- broaden path
- use search_text for literal candidates
```

### 8.3 parse-partial

```text
structural_search[find] q3
query: useEffect cleanup functions
language: typescript
scope: web/src; tests excluded
status: parse-partial
matches: 2 groups, 6 displayed matches, 8 known matches

Missing coverage:
- 2 files could not be parsed as TypeScript/TSX

[1] cleanup-returning useEffect calls
  web/src/App.tsx:44       useEffect(() => { ... return () => ... })

next:
- inspect displayed matches
- narrow path to parsed TSX files
```

### 8.4 invalid-pattern

```text
structural_search[find]
status: invalid-pattern
language: rust
pattern: match $POLICY { $$$ARMS

Problem
  Pattern cannot be parsed for the selected language.

next:
- explain_ast on a representative line
- simplify the pattern and retry
```

### 8.5 invalid-ref

```text
structural_search[expand]
status: invalid-ref
ref: q9.1

No registered structural_search result matches this reference in the current MCP connection.

next:
- use a visible qN.N reference from earlier structural_search output
- run a new find query
```

## 9. qN 引用状态

规则：

```text
成功查询分配 qN。
no-matches 查询分配 qN。
parse-partial 查询分配 qN。
参数错误不分配 qN。
invalid-ref 不分配 qN。
workspace-required 不分配 qN。
qN 只在当前 MCP connection 内有效。
ref="2" 指最近一次可展开查询的 [2]。
ref="q1.2" 指当前连接内 q1 的 [2]。
显式 qN.N 优先于最近查询。
```

派生查询成功后分配新的 qN，并更新 RecentQuery。

## 10. 实现分层

`docutouch-core` 推荐新增对象：

```text
StructuralSearchMode
StructuralSearchOptions
StructuralSearchResult
StructuralSearchSurfaceKind
StructuralSearchQuery
StructuralSearchRule
StructuralSearchMatch
StructuralSearchGroup
StructuralSearchRegistryEntry
```

`docutouch-server` 推荐新增对象：

```text
StructuralSearchArgs
StructuralSearchPathInput
StructuralSearchContextInput
StructuralSearchModeInput
connection-local registry holder
```

后端调用可以通过 ast-grep CLI 或 Rust library 适配。无论后端形态如何，core 对 server 暴露的结果必须是 DocuTouch 自有对象，不把 ast-grep 原始输出作为 public surface。
