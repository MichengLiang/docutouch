# structural_search 验收标准与评估矩阵

## 1. 验收对象

本文定义 `structural_search` 完成时的验收标准。验收对象是 DocuTouch MCP 工具的查询行为、输出表面、状态管理和测试覆盖。

验收不包含任何代码改写、替换、修复、autofix、rewrite 或 apply 行为。实现中出现这些入口即为不合格。

## 2. 总体验收标准

`structural_search` 达到可用状态必须同时满足：

```text
1. MCP tool schema 存在，参数与详细设计一致。
2. find、expand、around、explain_ast、rule_test 五个 mode 可调用。
3. ast-grep 查询对象覆盖 atomic、relational、composite、constraints/utility rules。
4. 默认输出为 pretty text。
5. 所有可注册结果使用 qN。
6. 结果组使用连续 [N]。
7. 证据行包含 path:line。
8. 大结果显示 omitted。
9. 所有输出显示 next。
10. invalid-ref、参数错误、workspace-required 不污染 RecentQuery。
11. parse-partial 显示缺口。
12. include_tests 选择可观察。
13. rule 中编辑字段被拒绝。
14. 黑盒测试覆盖 DES 中定义的基本流、异常流、等价类和边界值。
```

## 3. 功能验收矩阵

| 编号 | 能力 | 验收输入 | 必须观察到的输出 |
| --- | --- | --- | --- |
| F-01 | 简单 pattern 查询 | `find` + `pattern` + `language` + `path` | `structural_search[find] qN`、matches 统计、结果组、path:line、next |
| F-02 | rule object 查询 | `find` + `rule` | rule summary、匹配组、capture/context 摘要 |
| F-03 | atomic rule | `kind`、`pattern`、`regex`、`nthChild`、`range` fixture | 每类 rule 有成功 snapshot |
| F-04 | relational rule | `inside`、`has`、`precedes`、`follows` fixture | 输出显示结构关系摘要 |
| F-05 | composite rule | `all`、`any`、`not`、`matches` fixture | 输出显示 composite rule summary |
| F-06 | constraints | metavariable constraints fixture | 输出显示 constraints summary，匹配结果正确 |
| F-07 | utility rules | `matches` utility fixture | 输出显示 utils 名称，匹配结果正确 |
| F-08 | expand | `expand ref=1` | 输出 `from: qN.[M]`，展示更多 match/capture |
| F-09 | around | `around ref=qN.M` | 输出 Enclosing、Node、Siblings/Captures 或明确缺省 |
| F-10 | explain_ast | `explain_ast path:line` | 输出 source、language、node kind、local tree、next |
| F-11 | rule_test matched | 小范围 rule 命中 | 输出 status matched、matches、captures、next |
| F-12 | rule_test no-matches | 小范围 rule 不命中 | 输出 status no-matches、next |

## 4. 输出验收矩阵

| 编号 | 输出性质 | 验收条件 |
| --- | --- | --- |
| O-01 | pretty text 默认面 | 默认调用输出不是 JSON，不是 ast-grep 原始输出 |
| O-02 | 第一行 | 第一行格式包含 `structural_search[mode]`；可注册结果包含 qN |
| O-03 | 查询头 | 输出包含 pattern 或 rule summary、language、scope/path |
| O-04 | 统计 | success 输出包含 displayed/known match 或 group 统计 |
| O-05 | 结果组 | 结果组从 `[1]` 开始连续编号 |
| O-06 | 证据 | 每个实体 match 至少一条证据包含 `path:line` |
| O-07 | capture | capture 摘要短而可读；长 capture 有裁切或摘要 |
| O-08 | omitted | 结果超出展示预算时必须显示 omitted |
| O-09 | next | 每个输出包含 next；其中 ref 必须合法或明确是重跑建议 |
| O-10 | 派生来源 | expand/around/explain_ast 派生结果包含 `from:` 或 `source:` |
| O-11 | 错误短文本 | 错误输出不展示后端堆栈，不展示无关内部路径 |
| O-12 | tests 状态 | scope 行或 note 显示 tests included/excluded |

## 5. 状态验收矩阵

| 编号 | 状态行为 | 输入序列 | 验收条件 |
| --- | --- | --- | --- |
| S-01 | qN 顺序分配 | 连续三次成功 find | 输出 q1、q2、q3 |
| S-02 | no-matches 分配 qN | find 无匹配 | 输出 qN，RecentQuery 更新 |
| S-03 | invalid-pattern 不分配 qN | q1 成功后 invalid pattern，再 expand 1 | expand 仍指 q1.[1] |
| S-04 | invalid-ref 不分配 qN | q1 成功后 expand q99.1，再 expand 1 | 最后 expand 仍指 q1.[1] |
| S-05 | workspace-required 不分配 qN | 无 workspace 且无 path | 输出无 qN |
| S-06 | 显式历史引用 | q1=find A，q2=find B，expand q1.1 | 输出 from q1.[1] |
| S-07 | 隐式最近引用 | q1=find A，q2=find B，expand 1 | 输出 from q2.[1] |
| S-08 | connection 隔离 | 两个 MCP connection 各自产生 q1 | q1 各自指向本连接结果 |

## 6. 错误验收矩阵

| 编号 | 错误状态 | 构造方式 | 必须输出 |
| --- | --- | --- | --- |
| E-01 | workspace-required | 未设置 workspace 且 path 不可解析 | `status: workspace-required`、无 qN、next |
| E-02 | scope-error | path 不存在或不可读 | `status: scope-error`、无 qN、next |
| E-03 | ambiguous-language | 多语言目录且 language 缺失 | `status: ambiguous-language`、candidates、next |
| E-04 | unsupported-language | 显式传入不支持语言 | `status: unsupported-language`、next |
| E-05 | invalid-pattern | pattern 对 language 不可解析 | `status: invalid-pattern`、无 qN、next |
| E-06 | unsupported-rule-field | rule 含 fix/rewrite/autofix/apply | `status: unsupported-rule-field`、无 qN、next |
| E-07 | no-matches | 合法查询无结果 | `status: no-matches`、有 qN、next |
| E-08 | parse-partial | 部分文件解析失败 | `status: parse-partial`、Missing coverage、有 qN |
| E-09 | invalid-ref | ref 指向不存在 query/group | `status: invalid-ref`、无 qN、RecentQuery 不变 |

## 7. fixture 验收要求

实现必须包含可控 fixture workspace。fixture 应尽量小，避免直接依赖大型真实仓库才能触发测试。

### 7.1 Rust fixture

必须包含：

```text
函数调用。
match enum arms。
impl/function/module enclosing item。
测试文件和生产文件。
至少一个 relational rule 可命中的结构。
至少一个 composite rule 可命中的结构。
至少一个 no-matches 查询。
```

覆盖：

```text
pattern
kind
inside
has
precedes
follows
all
any
not
captures
include_tests
around
explain_ast
```

### 7.2 TypeScript fixture

必须包含：

```text
函数调用。
组件或普通函数。
*.test.ts 或 *.spec.ts 文件。
至少一个 AST 局部可用于 explain_ast。
```

覆盖：

```text
language inference
include_tests
pattern
parse-partial 可选构造
```

### 7.3 多语言 fixture

必须包含：

```text
Rust 文件。
TypeScript 文件。
Python 文件。
```

覆盖：

```text
ambiguous-language
显式 language 消除歧义
path 收窄消除歧义
```

### 7.4 大结果 fixture

必须构造超过默认 limit 的匹配集合。

覆盖：

```text
groups omitted
matches omitted
capture omitted
输出预算稳定
```

### 7.5 unsupported edit field fixture

必须构造包含编辑字段的 rule 输入。

覆盖：

```text
unsupported-rule-field
无 qN
RecentQuery 不变
```

## 8. 输出质量评估

输出质量不只由测试通过判定，还要人工检查以下项：

```text
1. 顶部查询头能在 5 行内说明本次查询对象。
2. 结果组标题表达结构共同点，而不是机械文件名堆叠。
3. 每组默认显示的证据足以决定下一步 read_file。
4. capture 摘要不淹没文件行证据。
5. omitted 说明能让 agent 判断是否需要收窄或展开。
6. next 建议是合法工具动作，不是泛泛说明。
7. 错误输出有下一步，不暴露后端堆栈。
8. parse-partial 不制造完整覆盖感。
9. tests included/excluded 状态容易看见。
10. 输出中没有编辑、修复、rewrite 动词作为工具行为。
```

## 9. 性能与预算验收

`structural_search` 是查询工具，必须保护工具返回体和执行时间。

验收要求：

```text
默认 preview 输出有固定最大 group 数。
每组默认有固定最大 evidence 行数。
每条 evidence 行有最大显示宽度。
长 capture 使用摘要或 inline omission。
极大结果不返回完整 match 数组。
超时应返回 unavailable 或 partial surface，而不是挂起。
```

建议初始预算：

```text
preview max groups: 8
preview max matches per group: 3
preview max capture samples per group: 3
```

实际数值由实现确定，但必须有测试锁定。

## 10. 集成验收

`docutouch-core`：

```text
新增 core 查询与渲染模块。
core 单元测试覆盖分组、输出、错误渲染。
后端输出不直接泄露到 public surface。
```

`docutouch-server`：

```text
新增 StructuralSearchArgs schema。
工具注册包含 structural_search。
call_json/call_mcp_tool 可分发 structural_search。
workspace/path 解析复用现有规则。
stdio smoke 覆盖 MCP 调用。
```

工具文档：

```text
docutouch-server/tool_docs/structural_search.md 存在。
工具描述与本设计一致。
描述中不出现编辑能力。
```

## 11. 回归风险清单

必须防止：

```text
1. 默认输出退化成 ast-grep JSON。
2. 大结果静默截断。
3. 结果组编号不连续。
4. next 引用不存在。
5. invalid-ref 更新 RecentQuery。
6. 显式 qN.N 被解析成最近查询。
7. no-matches 不分配 qN 或被写成参数错误。
8. invalid-pattern 分配 qN。
9. include_tests=false 仍展示测试命中且不说明。
10. parse-partial 不显示缺口。
11. path:line 丢失。
12. 长 capture 占满输出。
13. unsupported edit field 被执行或静默接受。
14. 多语言目录随机选择 language。
15. 后端错误堆栈进入默认输出。
```

## 12. 完成定义

`structural_search` 的完成定义：

```text
所有功能验收矩阵通过。
所有输出验收矩阵通过。
所有状态验收矩阵通过。
所有错误验收矩阵通过。
fixture 覆盖完整。
默认输出通过人工质量评估。
性能与输出预算有测试锁定。
server MCP smoke 测试通过。
工具文档与实现行为一致。
```

完成后，开发者可以用该工具在 agent 工作流中稳定完成以下任务：

```text
从文本候选进入 AST 结构候选。
按结构分组查看匹配。
展开匹配 capture。
查看匹配局部结构上下文。
解释 AST 局部形状。
在小范围测试 rule。
在异常和大结果情况下获得可继续行动的输出。
```
