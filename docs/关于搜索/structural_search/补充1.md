# DocuTouch `structural_search` 全套设计方案

## 0. 前提修正

你刚才指出的核心问题是对的：DocuTouch 不是为了把 `ast-grep` 的能力简单包一层，也不是为了做一个“比原生命令少几个字段”的轻量适配器。`ast-grep` 本身已经是成熟的结构化代码搜索后端；ChatGPT、Codex 或其他模型理论上完全可以直接调用原生 `ast-grep`，解析 JSON，自己决定下一步。但这件事在真实 agent 工作流里并不理想，因为原生输出不是为自回归、窗口有限、逐步决策的语言模型设计的。它包含大量机器友好的字段、局部意义不稳定的结构、后端内部形状、过多细节、以及需要二次解释的范围信息。模型可以读，但读得费上下文、费注意力，而且容易在一次响应里把“匹配对象”“候选 AST 节点”“行文本”“局部上下文”“下一步动作”混成一个东西。

所以 DocuTouch 的目标不应是“包装 ast-grep”，而应是“把 ast-grep 的结构化能力转换成适合模型连续推理的语义观察界面”。这里的关键不是最小节点、最大节点、候选顺序这种局部策略本身，而是工具输出必须形成稳定的认知契约：模型看到什么对象，就能知道它是什么、它从哪里来、它和后端匹配的关系是什么、它下一步该怎样缩放观察粒度。`structural_search around` 把 TypeScript 函数声明匹配显示成内部 `call_expression`，不是一个孤立 bug，而是契约层级混淆的症状：同一个 `Node` 区块里混合了“匹配文本”和“另一个候选 AST 节点的 kind/range”。这会让模型误判当前观察对象，进而破坏整个 `find -> around -> expand -> read_file -> edit` 的工作链。

下面的设计按这个定位展开：DocuTouch 是面向 agent 的代码观察层；`ast-grep` 是后端结构识别引擎；`structural_search` 的职责是将后端事实压缩成低噪声、可引用、可扩展、契约清晰的语义表面。设计重点不是追求某个单点策略最漂亮，而是建立一套对象层级、选择规则、输出语法、失败表面、测试准则和演进路径，使工具长期服务于模型，而不是把后端复杂度原样推给模型。

## 1. 产品定位

DocuTouch 的核心用户不是传统 CLI 用户，而是需要在代码库中持续观察、定位、理解和修改代码的语言模型。人类可以读长 JSON，可以回看大量终端输出，可以用经验补偿上下文噪声；模型则更依赖稳定的局部表面。模型每次调用工具后，都要把返回内容压缩进下一步推理，它不能无限保留后端细节，也不能可靠地区分“这个字段来自当前匹配”还是“这个字段来自一个启发式候选节点”，除非输出本身明确表达这些关系。

因此 DocuTouch 的第一原则是：输出不是后端日志，而是 agent 观察面。观察面必须将代码结构转化为模型可直接使用的任务信息。它不需要展示所有事实，但展示的每个事实都必须有明确对象归属。它可以省略后端字段，但不能让不同对象的字段看起来像来自同一对象。它可以压缩文本，但不能压缩掉身份、范围、来源、层级、引用路径和下一步动作。

`ast-grep` 的原生输出适合程序消费：它告诉调用方有哪些匹配、匹配文本是什么、范围在哪里、meta variables 是什么。DocuTouch 的输出适合模型消费：它应该告诉模型“你现在看的是哪个匹配对象”“这个对象属于哪类语法结构”“这个对象是否有更大的父结构和更小的子结构”“这个对象是否可安全作为编辑锚点”“如果你想看更多细节，应该 expand、around、explain_ast 还是 read_file”。两者的差异不是格式差异，而是语义责任差异。

所以设计目标可以表述为一句话：`structural_search` 将 `ast-grep` 的结构匹配结果重写为模型可稳定引用、可逐步展开、低噪声且对象边界明确的代码观察表面。

这句话里有四个不可删除的部分。第一，“结构匹配结果”说明 DocuTouch 不替代后端解析，不声称自己拥有完整 AST 解释权。第二，“重写为模型可稳定引用”说明输出不是原始事实堆叠，而是面向 agent 决策的表面。第三，“可逐步展开”说明工具必须支持从概览到局部到文件正文的渐进观察，而不是一次性塞满上下文。第四，“对象边界明确”说明每个字段必须归属到一个对象，不能把匹配对象和候选节点混写。

## 2. 核心问题重新定义

Issue #1 表面上是 TypeScript 函数声明匹配时，`around` 报告了内部 `call_expression`。但真正的问题不是“候选节点选小了”。如果只把问题定义为“选小了”，修复就会滑向调整候选顺序、改 `prefer_smallest`、按语言特判函数声明、或写一堆局部启发式。这样的修复会陷入你说的“最小不最小永远做不完”：今天函数声明被调用表达式抢走，明天 class 被 method 抢走，后天 Python decorator、Rust macro、JS arrow function、TS export wrapper 又会制造新的边界歧义。只盯着大小策略，工具会变成启发式补丁集合。

正确的问题定义应是：`around` 的输出没有区分“匹配对象”和“局部 AST 解释对象”。`find` 产生的是一个匹配对象，它有来源、范围、文本、语言、captures。`around ref=qN.M` 的默认语义应当围绕这个匹配对象展开，因为 ref 指向的是 find 的结果，而不是任意后端候选节点。局部 AST 解释可以帮助理解匹配对象，但它不能偷偷替换匹配对象。如果实现需要选择一个 AST 节点来给出 kind、range、children、pattern hint，这个节点必须先被定义为哪一类对象：是 matched node、covering node、enclosing node、nearest child node，还是 representative node。没有这个对象定义，任何大小策略都可能在某个样例上看似合理，在另一个样例上变成误导。

因此 issue #1 的契约级根因是：输出表面把多个对象层级投影到了一个 `Node` 区块，却没有声明字段来源。`text:` 来自匹配对象；`kind:` 和 `range:` 来自内部候选节点；`captures:` 又来自匹配对象。模型看到这个组合，会自然推断这些字段描述同一个对象。这种推断是合理的，因为工具表面就是这样写的。问题不在模型理解能力不足，而在工具输出违反了自己的语义表面。

这意味着修复目标不能是“让这个 fixture 显示 function_declaration”这么窄。修复目标应是：`around` 对引用目标始终先呈现匹配对象本身；任何辅助节点必须明确标注其关系；当后端能够找到与匹配范围对应的 AST 节点时，将它作为 matched node；当找不到时，也不能把 fallback 节点伪装成 matched node。

## 3. 对象模型

为了让 `structural_search` 长期稳定，必须定义它输出中的核心对象。没有对象模型，字段会继续凭感觉摆放。

第一个对象是 `StructuralMatch`，也就是 ast-grep 搜索产生的匹配对象。它的身份由 `query id + group index + match index` 或当前简化表面中的 `qN.[group]` 推导。它至少包含文件、起始行列、结束行列、语言、匹配文本、上下文行、captures。它是 `find`、`expand`、`around ref`、`rule_test ref` 的主对象。模型使用 ref 时，默认引用的就是这个对象。

第二个对象是 `MatchedAstNode`，也就是与 `StructuralMatch` 在 AST 层对应的节点。它必须满足强关系：范围相同、文本相同或在可解释的 wrapper 差异内等价。对于 TypeScript `export function foo(){}`，ast-grep pattern 可能返回不含 `export` 的函数声明文本，而文件行含有 export wrapper；这里就需要明确坐标和文本归一规则。只要能证明节点对应的是匹配对象本身，就可以把它作为 `Node.kind` 的来源。否则它不能进入主 `Node` 区块。

第三个对象是 `EnclosingAstNode`，即包含匹配对象的外层语法结构。它通常用于告诉模型“这个调用位于哪个函数/impl/class/method 中”。它不应与主节点混淆。比如调用表达式的匹配对象可以有 `Node.kind = call_expression`，同时 `Enclosing.item = function_item pub fn run`。这两个对象层级不同，但都对模型有价值。

第四个对象是 `InnerAstNode` 或 `NearestChildNode`，即匹配对象内部的子结构。当匹配对象是函数声明、类声明或块级结构时，它内部可能包含调用、返回语句、变量声明等。内部节点可以作为 `Children` 或 `Notable children` 输出，但不能替代匹配对象。它的价值是辅助模型决定下一步是否需要更细粒度搜索，而不是定义当前 ref 的身份。

第五个对象是 `RepresentativeLineTarget`，用于 `explain_ast path:line` 这种没有 ast-grep 匹配对象的入口。它不是搜索结果，只是一个“行位置观察请求”。这一类目标可以有不同的选择策略，因为用户没有提供具体匹配范围，只提供了一行。对它来说，工具可以选择行上的主导结构、最小有意义节点、或一组候选节点。但必须在输出里说明这是 line target 的 representative node，而不是 matched node。`explain_ast` 的设计要和 `around ref` 分开，不能把两者的策略硬混成一个函数。

第六个对象是 `ObservationSurface`，也就是最终给模型看的文本。它不是 AST，也不是 JSON dump。它负责把上述对象投影成稳定、少噪声、可继续调用的文本。它的设计标准不是“信息最多”，而是“下一步决策成本最低”。

## 4. 模式语义

`find` 的语义是产生一组结构匹配对象，并把它们按对模型有用的类别分组。它应优先显示每组的代表性证据、captures 摘要、总量、隐藏量和下一步动作。它不应展示完整 AST，也不应混入太多后端实现细节。`find` 的输出目的是让模型判断是否命中了正确对象，以及是否需要 `expand` 或 `around`。

`expand` 的语义是展开某个组内的匹配对象列表。它仍然以 `StructuralMatch` 为主对象。它应显示更多匹配实例、完整一些的 captures、上下文摘要和 omission accounting。`expand` 不应偷偷改变对象身份，也不应开始解释 AST 层级；如果需要局部结构，交给 `around`。

`around` 的语义是围绕一个具体匹配对象展示局部结构。这里的主对象必须是 ref 指向的 `StructuralMatch`。`around` 可以派生出 `MatchedAstNode`、`EnclosingAstNode`、`Children`、`Siblings` 和 `Captures`，但它们都要围绕主对象服务。主 `Node` 区块必须描述匹配对象本身；如果不能建立 AST 对应关系，应输出 `kind: inferred` 或 `kind: unknown/syntax_node`，并明确 `node_source: match-range fallback`，而不是拿内部节点冒充。

`explain_ast` 的语义是解释一个位置或引用的 AST 形状。当它接受 ref 时，它应尽量沿用 `around` 的匹配对象语义，因为 ref 指向的是匹配对象。当它接受 path:line 时，它应承认这是位置观察，不是匹配观察。对 path:line，建议输出 `Line target` 与 `Candidate nodes` 两层：先显示源行，再列出覆盖该行的候选节点，按关系排序，例如 `starts_on_line`、`contains_line`、`smallest_named_on_line`。如果为了简洁只显示一个 node，也应命名为 `Representative node`，而不是让模型误以为这是唯一 AST 真相。

`rule_test` 的语义是把一个小源片段作为规则试验场。它的输出目标不是完整搜索，而是快速判断 pattern/rule 是否能在局部源上命中。它应保持输入目标和命中结果的区别：test source 是一段源，matched result 是 rule 在这段源里的匹配。

## 5. `around` 的输出契约

`around` 是这次问题的中心，因为它连接了搜索结果和局部理解。它应该有一个明确契约：`around ref=qN.M` 的主节点是 ref 指向的匹配对象；所有结构解释都必须声明与主节点的关系。

推荐输出骨架如下：

```text
structural_search[around] q10
from: q7.[1] production matches
source: projects/.../helpers.ts:3

Matched item
  language: TypeScript
  kind: function_declaration
  range: 3:8-8:2
  text: function codexPathOverride() { return (...); }
  node_source: exact ast-grep kind/range match

Enclosing
  item: module/file

Siblings
  previous: import path from "node:path";
  next: none

Children
  return_statement: return (...)
  call_expression: path.join(...)
  call_expression: process.cwd()

Captures
  $F = codexPathOverride
  $$$ARGS =
  $$$BODY = return (...)

next:
- expand 1       show all matches in this group
- read_file around the source line
- around child 1  inspect a notable child node when supported
```

这里最重要的是把 `Node` 改成 `Matched item` 或至少让 `Node` 明确是 matched item。`kind` 的来源必须是对应匹配对象的 AST 节点，而不是任意重叠候选。`Children` 可以展示内部 `call_expression`，但它的对象地位是 child，不是 current node。模型看到这个表面，就不会把函数声明误解成调用表达式，同时仍能发现函数体里有 `path.join` 和 `process.cwd()` 这些值得继续搜索的节点。

如果当前实现不准备引入 child ref，那么 `Children` 可以先保持摘要，不需要提供可引用子节点。关键是语义层级要对：主对象在上，内部对象在下，fallback 关系可见。

如果 AST 节点无法精确对应匹配对象，输出应类似：

```text
Matched item
  language: TypeScript
  kind: syntax_node
  range: 3:8-8:2
  text: ...
  node_source: match range; no exact AST kind candidate found

Nearest AST candidates
  enclosing: function_declaration ...
  inner: call_expression process.cwd()
```

这个设计承认不确定性，而不是伪造确定性。对模型来说，诚实的不确定性比错误的确定性更有价值。

## 6. AST 选择策略

选择策略应从对象契约倒推，而不是从“大小”倒推。`around ref` 的目标是匹配对象，因此选择顺序应是：先找对应匹配对象的 AST 节点，再找覆盖匹配对象的外层节点，再找匹配对象内部的显著子节点。内部子节点不应进入主节点槽位。

第一阶段是 exact candidate。候选节点满足以下条件之一，可以认为对应当前匹配：范围完全相同；文本归一后完全相同且范围包含匹配范围；在语言 wrapper 规则下等价，例如 TypeScript `export function` 的 pattern 匹配文本从 `function` 开始，而 AST 节点范围可能从 `function` 开始或包含 export modifier，具体取决于 tree-sitter/ast-grep 返回。这里不能靠单一字符串 contains 判断，必须同时考虑范围关系、文本关系和节点类别。

第二阶段是 covering candidate。它表示 AST 节点覆盖当前匹配，但不一定等于当前匹配。这个阶段适用于匹配文本是函数签名的一部分、语句片段、表达式片段等场景。covering candidate 可以作为 `Enclosing` 或 `Covering node` 输出，但如果它比匹配对象大很多，不应无条件作为主节点。

第三阶段是 inner candidate。它表示候选节点位于匹配对象内部。这个阶段对概览很有帮助，尤其是函数、类、impl、方法体内部。但它的语义是 child/notable inner node。它绝不能因为 score 小就覆盖主节点。当前 issue 正是把第三阶段的对象放进了第一阶段的位置。

第四阶段是 fallback inference。无法从 ast-grep kind scan 建立关系时，根据语言和文本做轻量推断。这个 fallback 应该被明确标注，避免模型把它当成后端 AST 事实。比如 `kind: function_declaration` 可以来自文本 `function run()` 的推断，但应有 `node_source: inferred from match text`。如果不想增加字段，也至少不要显示错误 range。

这个策略里，“最小”不是消失，而是降级到特定语境：当匹配对象本身就是一个表达式或行位置没有明确匹配对象时，最小有意义节点可以是好的 representative；当匹配对象是大结构时，最小内部节点是 child，不是主节点。这样就不会陷入“永远调最小/最大”的局部战争。

## 7. `explain_ast path:line` 的重新设计

`explain_ast path:line` 和 `around ref` 不应共享同一个未区分语义的 `find_ast_node(item)`。它们的输入对象不同。

`around ref` 有匹配对象，应该围绕匹配对象解释。`explain_ast path:line` 只有文件和行号，应该解释这行在 AST 中的候选结构。它不能假装有一个唯一 matched node。

推荐 `explain_ast path:line` 输出如下：

```text
structural_search[explain_ast] q13
source: projects/.../helpers.ts:3
language: typescript

Line target
  text: export function codexPathOverride() {

Candidate nodes
[1] starts_on_line
  kind: function_declaration
  range: 3:8-8:2
  text: function codexPathOverride() { ... }
  pattern hint: function $F($$$ARGS) { $$$BODY }

[2] enclosing
  kind: program
  range: 1:1-8:2
  text: import path ...

[3] inner_on_line
  kind: identifier
  range: 3:17-3:34
  text: codexPathOverride

next:
- structural_search find with candidate pattern [1]
- read_file around source line
```

如果为了上下文预算只显示一个候选，也应该写：

```text
Representative node
  relation: starts_on_line
  kind: function_declaration
```

这样模型知道这是位置代表节点，不是搜索匹配对象。对于 issue 中的 `explain_ast`，行 3 开始函数声明，输出 `call_expression` 显然不符合用户直觉，也不符合模型需要。即使内部有 `process.cwd()`，它在第 6 行，不应该成为第 3 行的 representative node。这里的策略应优先 `starts_on_line` 的命名结构，其次覆盖该行的最小命名结构，再其次内部节点。跨多行内部调用不应抢占函数声明起始行。

这个设计会让 `explain_ast` 成为 pattern 发现工具，而不是随机局部节点选择器。模型使用它时，通常是在不知道 pattern 怎么写的情况下询问“这一行是什么结构”。因此它应该给出候选 pattern hints，帮助模型构造下一次 `find`，而不是只给一个可能误导的 kind。

## 8. 输出表面设计原则

DocuTouch 的输出应该遵守六个表面原则。

第一，主对象先行。每个模式都必须先显示当前主对象。`find` 的主对象是 group/match；`expand` 的主对象是 match list；`around ref` 的主对象是 matched item；`explain_ast path:line` 的主对象是 line target；`rule_test` 的主对象是 test source 和 rule result。

第二，字段来源可判定。`kind`、`range`、`text`、`captures` 不能来自不同对象却放在同一对象块里。如果确实需要混合，必须拆块或加来源字段。模型不是不能理解复杂关系，而是不能被迫从不一致字段里猜关系。

第三，默认输出服务下一步动作。每个输出都应该让模型知道下一步最可能是什么：expand、around、read_file、rule_test、broaden path、narrow path、change language、fix pattern。下一步动作不是帮助文案，而是 agent 控制流的一部分。

第四，压缩要有损但不能失真。DocuTouch 可以截断长文本、隐藏额外匹配、只展示代表项，但不能把 A 对象的 kind 安到 B 对象上。上下文预算允许省略，不允许错配。

第五，渐进披露必须保持引用稳定。`qN`、group index、match index、source path、line range 是模型跨工具调用的锚点。任何输出变化都不应让模型失去回到原对象的路径。未来如果支持 child refs，也必须显式分配新的引用域，而不是复用 group ref。

第六，失败表面要可行动。`invalid-ref`、`unsupported-language`、`parse-partial`、`invalid-pattern`、`scope-error` 都应给出模型下一步可执行动作。失败不是异常日志，而是观察计划的分支点。

## 9. 结构层级的推荐输出词汇

为了避免“node”一词承载太多含义，建议建立稳定术语。

`Matched item`：ref 或搜索结果对应的匹配对象。用于 `around ref` 和 `expand`。

`Matched AST node`：与 matched item 对应的 AST 节点。只有在关系可证明时使用。

`Representative node`：位置查询的代表节点。用于 `explain_ast path:line`。

`Enclosing node`：包含当前对象的外层结构。

`Inner nodes` 或 `Notable children`：当前对象内部的显著节点。

`Candidate nodes`：没有唯一主节点时的候选列表。

`node_source` 或 `relation`：说明节点与当前目标之间的关系，例如 `exact_range`、`same_text`、`covers_match`、`inside_match`、`starts_on_line`、`contains_line`、`inferred_from_text`。

这些词不是装饰，而是契约工具。模型会把它们当成类型标签使用。只要术语稳定，模型就能形成固定策略：看到 `Matched item` 就知道这是编辑锚点；看到 `Enclosing node` 就知道这是上下文；看到 `Inner nodes` 就知道这是潜在的下一轮搜索对象；看到 `Representative node` 就知道这是位置解释，不一定是搜索结果。

## 10. 实现架构

当前实现里 `LocalSyntaxContext::build(item)` 承担了太多责任：读源文件、选择 AST 节点、推断 kind、找 enclosing、找 siblings、推断 children、生成 pattern hint、计算 range。它把对象解析和输出需求混在一起，导致 `around` 和 `explain_ast` 共享了一个不够明确的局部上下文对象。

建议把内部结构拆成三层。

第一层是目标解析层：`ResolvedTarget` 继续表示用户输入解析结果，但要区分 target kind。可以引入 `TargetKind::MatchRef`、`TargetKind::PathLine`、`TargetKind::LineOnlyWithSearchFile`。这样后续逻辑知道自己面对的是匹配对象还是位置对象。

第二层是 AST 关系层：新增一个 `AstRelationResolver` 或一组纯函数，输入源文件、语言、目标范围、目标文本，输出 `AstRelationSet`。这个集合包含 `matched_node`、`covering_nodes`、`enclosing_nodes`、`inner_nodes`、`line_candidate_nodes`。每个节点都带 `relation` 和 `confidence/source`。它不负责格式化文本，也不决定最终显示多少。

第三层是 surface 格式化层：`format_around_result`、`format_explain_ast_result` 根据模式语义选择关系集合里的对象并生成文本。`around` 使用 `matched_node` 作为主节点；没有 matched node 时使用 match range fallback。`explain_ast path:line` 使用 `line_candidate_nodes` 生成候选列表或 representative node。

这种拆分的好处不是“抽象更漂亮”，而是避免语义错位。选择 AST 关系是一件事，格式化给模型看是另一件事。只有分开，才能在测试里直接验证：某个 TypeScript 函数声明的 relation set 中，`function_declaration` 是 `matched_node`，`process.cwd()` 是 `inner_node`，而不是只在字符串输出里猜。

## 11. 节点关系判定细节

节点关系判定需要同时使用范围和文本，不能只用 `contains`。

范围应使用统一坐标系。当前 `StructuralMatch` 用 1-indexed line/column，ast-grep raw range 是 0-indexed line、column optional。内部应统一成一个 `SourceRange` 类型，明确 start inclusive、end exclusive 或 display end 的规则。输出可以继续使用用户友好的 `line:column-line:column`，但内部比较不能混用显示坐标。

文本归一应区分 display text 和 comparison text。display text 可以折叠空白、截断宽度；comparison text 不应使用截断后的文本。当前 `StructuralMatch.text` 已经经过 `trim_text`，长文本可能被截断，这不适合做精确关系判断。应保留 raw match text 或至少在 `StructuralMatch` 里增加未截断文本字段，例如 `raw_text`、`raw_lines`。如果没有 raw text，关系判断会因为截断而退化。

关系判定可以按以下顺序：

1. `exact_range`: candidate range 与 match range 相同。
2. `same_normalized_text`: candidate text 与 raw match text 归一后相同。
3. `candidate_covers_match`: candidate range 包含 match range，且 candidate text 的归一文本包含 match raw text。
4. `match_covers_candidate`: match range 包含 candidate range，且 match raw text 包含 candidate text。
5. `line_relation`: candidate 与 path:line 的关系，例如 starts_on_line、contains_line、ends_on_line。

对 `around ref`，主节点只接受 `exact_range` 或 `same_normalized_text`，必要时接受受控的 `candidate_covers_match`，但必须标注 relation。`match_covers_candidate` 应归入 inner nodes，不应作为主节点。

这正好解释 issue #1：函数声明匹配的 raw match 覆盖整个函数，内部 `process.cwd()` 满足 `match_covers_candidate`，但不满足 matched node 关系。因此它只能是 inner node，不能是主节点。

## 12. 候选 kind 表设计

候选 kind 表不应承担语义选择责任。它只定义“有哪些节点值得询问 ast-grep”。当前 TypeScript/JavaScript 表把 `call_expression` 放在前面，然后选择器又用全局 smallest 策略，导致顺序和大小共同影响语义。未来应让 kind 表只负责覆盖面，关系 resolver 负责选择。

候选表可以按语言分组：

TypeScript/JavaScript：`function_declaration`、`method_definition`、`class_declaration`、`lexical_declaration`、`variable_declarator`、`arrow_function`、`call_expression`、`return_statement`、`import_statement`、`export_statement` 等。是否全部加入要看 ast-grep/tree-sitter 的实际 kind 名称和性能成本。

Rust：`function_item`、`impl_item`、`struct_item`、`enum_item`、`trait_item`、`mod_item`、`macro_invocation`、`call_expression`、`match_expression`、`match_arm`、`let_declaration` 等。

Python：`function_definition`、`class_definition`、`call`、`assignment`、`return_statement`、`import_statement` 等。

但是输出不应暴露“我们查了哪些 kind”作为主信息。候选表是后端实现细节。模型需要的是稳定关系：matched、enclosing、inner、candidate。

## 13. 性能与缓存

`around` 和 `explain_ast` 当前通过多次 `ast-grep scan --inline-rules kind=...` 查询候选 kind。这个方式简单，但随着候选 kind 增加会变慢。因为 DocuTouch 是 agent 工具，性能不只是响应时间，还影响模型工作节奏。每次工具调用都应尽量在可预测时间内返回。

短期可以保留多次 kind scan，但要加两个约束。第一，只对目标文件执行，不对目录重复扫描。第二，候选 kind 表不要盲目扩张；围绕主语言和高价值结构补齐。对于 issue #1，修复不需要大规模候选扩张。

中期可以引入 per-file AST kind cache。缓存 key 包含 path、mtime、language、kind set。一次 `around` 或 `explain_ast` 可以批量收集候选节点，然后在内存里做关系排序。Rust 测试中用 tempdir，缓存要可关闭或自然隔离。

长期可以考虑让 ast-grep 一次 rule 扫多个 kind，或使用 ast-grep JSON/规则组合减少进程启动次数。但这不是第一阶段核心。先把语义契约修对，再优化后端调用。

## 14. 测试策略

测试应覆盖契约，而不是覆盖某个候选顺序。

第一类测试是 issue 回归测试：TypeScript 文件中有 `export function codexPathOverride()`，函数体里有 `path.join(process.cwd(), ...)`。先 `find` pattern `function $F($$$ARGS) { $$$BODY }`，再 `around ref=q1.1`。断言输出包含 `kind: function_declaration`，不把主节点显示为 `call_expression`。同时可以断言 `Captures` 仍包含 `$F = codexPathOverride`。如果设计输出 `Inner nodes`，再断言 `process.cwd()` 只出现在 inner/children 区块，而不是主 kind。

第二类测试是普通调用不回退：Rust 或 TypeScript 中搜索 `evaluateExecPolicy($$$ARGS)`，`around` 仍报告 `call_expression`。这防止修复函数声明时把所有东西都推向外层函数。

第三类测试是 `explain_ast path:line`：对 TypeScript 函数声明起始行，代表节点或候选首项应是 `function_declaration`，不能是后续行内部调用。对调用表达式所在行，代表节点应是 `call_expression` 或候选中明确包含调用。这样区分 line-start structure 和 expression line。

第四类测试是对象字段一致性：当输出一个 `Matched item`，它的 `range` 应来自匹配对象或 matched AST node，不能来自 inner node。可以通过构造跨多行函数匹配和内部单行调用来断言 range 包含函数开始行和结束行。

第五类测试是 fallback 测试：当没有 source_path 或 ast-grep 不可用时，`LocalSyntaxContext` 仍能推断 kind，但输出不应声称 exact AST node。现有 `local_context_falls_back_without_source_backing` 可以保留并调整语义。

第六类测试是文档/工具说明测试：`tool_docs/structural_search.md` 应明确 `around` 的 node 语义，避免未来实现回到“nearest inner node”。如果工具说明是给模型看的，文案本身就是接口的一部分，也应该被 review。

这些测试共同约束“对象边界明确”这个设计目标，而不是只约束某个例子。

## 15. 文档与工具说明

`structural_search` 的工具文档应该从“有哪些模式”升级为“每个模式返回什么观察对象”。模型读工具描述时，最需要知道的是输入对象和输出对象的关系。

`around` 文档建议写成：

```text
around：围绕 ref 或 path:line 目标展示局部结构。ref 目标的主对象是已注册的 structural match；Node/Matched item 描述该匹配对象本身。Enclosing 描述包含该匹配对象的外层结构。Children/Inner nodes 描述匹配对象内部的显著结构。path:line 目标没有 registered match；输出描述该行的代表结构或候选结构。
```

这段文字不应说“当前实现会怎样 fallback”，而应定义契约。实现达不到时要么修实现，要么输出 compatibility note，而不是让文档迁就 bug。

`explain_ast` 文档建议写成：

```text
explain_ast：解释 ref 或 path:line 的 AST 观察面。ref 输入沿用匹配对象语义；path:line 输入以源行为目标，返回覆盖该行或从该行开始的候选结构，并给出可用于下一次 structural_search 的 pattern hints。
```

这样模型就知道它不是精准搜索结果，而是 pattern 发现辅助工具。

## 16. 对这次 issue 的具体修复路线

如果按完整设计落地，不建议一口气重写整个 `structural_search`。这是一个长期实验工作台，但仍应控制改动风险。可以分三步。

第一步，修复契约破坏点。新增 TypeScript 回归测试，确认当前 `around` 把函数声明显示成 `call_expression`。实现最小但语义正确的 resolver：`find_ast_node_for_match` 先查 exact/same-text/covers-match 的结构节点，拒绝 match-covers-candidate 作为主节点。保留现有 enclosing 和 children 推断。这个阶段目标是 issue #1 不再误导。

第二步，拆分 `around ref` 和 `explain_ast path:line`。为 path:line 增加 representative/candidate 语义，避免函数声明起始行被内部调用抢走。这个阶段可以不做完整 candidate list，但要把输出命名从“local tree 的唯一 node”改为更准确的 surface。

第三步，整理内部对象模型。引入 `SourceRange`、保留 raw match text、建立 `AstNodeRelation`，把格式化和关系判定拆开。这个阶段会让代码更稳，也为后续 child refs、缓存、更多语言支持打基础。

如果当前用户期待的是快速关闭 issue，可以先做第一步和相关文档；如果目标是把 DocuTouch 做成“极致模型观察层”，则第二、第三步应进入近期计划。

## 17. 为什么不采用“只调最小/最大”策略

“最小节点”策略适合表达式搜索。用户搜 `foo($$$ARGS)`，模型希望看到的就是调用表达式，不希望看到外层函数。此时最小命名节点通常降低噪声。

“最大节点”策略适合上下文理解。用户搜某个调用后，模型可能需要知道它在哪个函数或类里。此时外层节点有价值，但它是 enclosing，不是 matched。

函数声明、类声明、impl 块、方法定义这些大结构搜索需要的是“匹配对象本身”。它既不是内部最小节点，也不一定是外层最大节点。它是 ref 指向的结构对象。

所以全局最小或全局最大都不是正确抽象。正确抽象是关系：matched、enclosing、inner、representative。大小只在同一关系内用于排序。比如多个 inner node 中可以优先显示最小或最显著；多个 enclosing node 中可以优先显示最近外层；多个 line candidates 中可以按 starts_on_line、named structure、span 大小排序。但跨关系比较大小没有意义。

这就是“见森林”的地方：森林不是节点大小，而是 agent 如何从一个观察对象移动到另一个观察对象。工具应该提供认知地图，而不是只返回一个树节点。

## 18. 面向模型的极致标准

所谓“做到极致”，不是输出更多，也不是把 ast-grep 的 JSON 改成漂亮文本。极致是让模型每次观察后减少不确定性。

模型最怕三类不确定性。第一是对象不确定：我现在看的到底是匹配、父节点、子节点还是某一行？第二是范围不确定：这个 range 能不能作为编辑锚点？第三是下一步不确定：我应该 expand、around、read_file 还是换 pattern？

DocuTouch 的结构化搜索输出应该逐项消除这些不确定性。每个模式都要回答：当前对象是谁；它的范围是什么；它的结构身份是什么；这个身份来自后端 AST、文本推断还是 fallback；它和附近结构的关系是什么；如果我需要更大视野或更小视野，下一步怎么做。

这套标准比“少噪声”更高。少噪声只是压缩；消除不确定性才是 agent 工具设计。一个短但混淆对象的输出，比一个稍长但对象清楚的输出更差。Issue #1 的输出不长，但非常伤害模型，因为它让模型怀疑 ref 是否错了、pattern 是否错了、工具是否错了。这种怀疑会导致额外工具调用、重复验证、甚至错误修改。

## 19. 兼容性与迁移

现有输出已经被测试和可能的用户习惯依赖。改动时要避免无意义破坏。

短期可以保留 `Node` 标题，但改变内容契约：`Node` 描述 matched item。也可以加一行 `node_source:`，这会增加输出但增强语义。如果担心破坏测试，先更新测试按关键语义断言，而不是完整字符串断言。

如果改成 `Matched item`，语义更清楚，但输出表面变化更大。可以分阶段：第一阶段 `Node` 下加 `relation: matched item` 或 `node_source`；第二阶段文档宣布 `Matched item`；第三阶段再调整标题。

`explain_ast` 的 path:line 输出变化可能更明显。这里建议先做不破坏性增强：保留 `node kind`，但把选择策略修成 line-start/line-contained 正确；之后再加 `Candidate nodes`。这样可以先解决误导，再演进表面。

## 20. 验收标准

这套设计的验收不应只看 issue 是否关闭，而应看 agent 工作链是否更稳。

最低验收：TypeScript 函数声明匹配的 `around` 主节点显示 `function_declaration`，range 对应函数声明匹配，不显示内部 `call_expression` 作为主 kind。普通调用匹配仍显示 `call_expression`。相关测试通过。

契约验收：`around ref` 的主对象字段来源一致；captures 与 matched item 对齐；enclosing 和 inner 信息不会伪装成主节点。工具文档明确 ref 语义。

模型验收：给模型一个 `find -> around` 输出，它不需要额外验证就能判断当前 ref 是函数声明，并能选择 `read_file` 或继续搜索内部调用。给模型一个 `explain_ast path:line` 输出，它能从函数声明行得到可用 pattern hint，而不是被内部调用带偏。

工程验收：关系判定逻辑有单元测试；TypeScript/Rust/Python 至少各有一个代表性测试；输出格式测试不依赖脆弱全文匹配；`cargo test -p docutouch-core structural_search` 和必要 workspace 测试通过。

## 21. 最终建议

我建议把 DocuTouch 的 `structural_search` 明确定义为“面向语言模型的结构观察层”，而不是“ast-grep wrapper”。在这个定位下，issue #1 的修复不是局部排序问题，而是对象契约修复。`around ref` 必须围绕 matched item；内部 AST 节点只能作为 child/inner 信息；`explain_ast path:line` 必须承认自己是位置观察，并给出代表结构或候选结构。

具体实施上，先用回归测试锁住 TypeScript 函数声明场景，再修改节点选择逻辑，阻止内部节点覆盖匹配对象。随后把 `around` 与 `explain_ast` 的目标语义拆开，逐步引入 `AstRelationSet`、`SourceRange`、raw text 保留和 candidate relation。这样既能优雅解决当前 issue，也能避免未来在“最小/最大”之间反复打补丁。

这才符合 DocuTouch 的真实价值：不是替模型调用 ast-grep，而是替模型把 ast-grep 的结构事实整理成可观察、可引用、可继续行动的语义表面。
