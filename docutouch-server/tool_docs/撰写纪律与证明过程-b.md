# `apply_patch` B 区构造证明文档（审核存档稿）

## 文档性质

本文档不是工具正文，不是教程，不是调用入口说明，也不是面向终端使用者的帮助页。  
本文档是一份用于团队审核与留档的**构造证明文档**。其职责不是再次定义 `apply_patch` 的工具本体，而是证明：在 A 区正文已经给出工具本体、输入边界、执行语义与失败结构的前提下，为什么仍然需要在其后接续一组面向理解兑现的 B 区正文；为什么 B 区必须这样构造；为什么 B 区不能退化为 A 区的重复，也不能膨胀为杂项说明层；以及 B 区各分面的必要性、不可合并性、删除后损失与写作纪律。

本文档的审核对象不是“文风是否顺滑”，而是下列问题：

1. B 区是否具有独立而必要的语义职责；
2. B 区是否始终围绕 A 区运作，而不改写 A 区；
3. B 区的各模块是否按职责拆分，而非按表面题材堆叠；
4. B 区是否优先处理高代价误读，而非无序增加说明；
5. B 区是否能够提高模型的理解兑现率，而不削弱 A 区的边界稳定性。

# 第一部分：顶层构造原则

## 0. 总公理表

| 项目 | 内容 |
| --- | --- |
| 对象 | `apply_patch` 工具文档中的 B 区正文 |
| 上位前提 | A 区正文已经完成工具本体、输入边界、语法结构、执行语义、路径规则、兼容性偏差与失败表面的定义 |
| 目标读者 | 团队审核者、构造者、维护者 |
| B 区正文目标读者 | 大语言模型 |
| 文本类型 | 面向理解兑现的受控阐释结构 |
| 主要职责 | 降低高代价误读；建立进入路径；提供失败后的重锚顺序；帮助模型把局部经验回收到 A 区 |
| 非职责 | 重新定义工具本体；发明第二套语法；解释后台构造学；陈述作者自我动作；记录维护流程 |
| 核心约束 | 忠实于 A；不新增本体承诺；优先处理高风险误读；前台可连续成文，后台必须可分析分层 |
| 评价标准 | 必要性测试、不可合并性测试、删除后损失测试、越界测试、回收性测试、排序合理性测试 |

## 0.1 本文档为何必须存在

A 区的构造证明已经证明了工具本体定义为何必须独立、为何必须保持术语控制、边界隔离、运行语义诚实与失败对象结构化。  
但 A 区的成立，并不自动推出“模型将以正确方式接收 A 区”。A 区负责概念保真，B 区负责理解兑现。二者不是风格差异意义上的两个版本，而是职责不同、合法性来源不同的两个层。若缺少 B 区构造证明，团队在审核 B 区时极易退回两类错误直觉：

1. 误将 B 区理解为“把 A 说得更通俗一点”；
2. 误将 B 区理解为“可以无限追加的补充说明区”。

前者会导致 B 区被压缩回 A 区重复；后者会导致 B 区膨胀成无边界的杂物层。  
因此，B 区必须有独立的证明文档，以锁定其合法性边界与结构纪律。

## 0.2 B 区的合法性基础

B 区成立的前提不是“A 区不够完整”，而是“接收者不会自动按理想路径处理 A 区”。  
因此，B 区的合法性来自如下事实：

- A 区可以在逻辑上成立，而模型仍可能在接收层失败；
- 模型的失败并不总体现为“看不懂”，更常体现为“过早自以为懂了”；
- 局部误读一旦发生，往往会污染后续全部补丁生成与失败归因；
- 说明资源并非无限，因此 B 区必须排序，而非穷举所有可能说明。

由此，B 区不是可有可无的装饰层，而是 A 区在真实使用中得以兑现的必要层。

# 第二部分：B 区整体结构证明

## 1. B 区为何不能写成“第二份 A 区”

| 维度 | 说明 |
| --- | --- |
| 被拒绝的误解 | 将 B 区理解为“更易懂版本的 A 区” |
| 错误原因 | 这会把 B 区退化为重复定义，既浪费篇幅，又无法处理真实误解路径 |
| A 区职责 | 定义工具本体、输入边界、运行语义与失败结构 |
| B 区职责 | 建立进入路径、前置拦截高风险误读、规定失败后的重锚顺序、帮助模型将局部经验回收到 A 区 |
| 结论 | B 区不是 A 区的重写本，而是围绕 A 区运行的理解兑现层 |

**证明**：  
如果 B 区只是把 A 区换一种说法再复述一遍，那么它只能增加篇幅，不能增加判别力。真正需要 B 区的地方，不在“再说一遍规则”，而在“模型最容易在哪里误读这些规则、误读后如何回到正确轨道”。因此，B 区的单位组织原则不能是“哪条规则值得再解释”，而应是“哪种误解最值得优先拦截”。

## 2. B 区为何不能写成 FAQ 或补充说明仓库

| 维度 | 说明 |
| --- | --- |
| 被拒绝的误解 | 将 B 区理解为“可持续追加的问答区” |
| 错误原因 | FAQ 组织按表面问题聚合，无法保证语义职责单一，也无法保证优先级合理 |
| 结构风险 | 容易把低价值补充、作者辩护、偶发案例、接入层细节混入正文 |
| 审核后果 | 文档不断增长，但模型一次性正确使用工具的概率未必提升 |
| 结论 | B 区必须采用分面结构，而不能采用问题堆叠结构 |

**证明**：  
FAQ 适合应对离散用户问题，不适合承担稳定的受控语义职责。B 区的目标不是回答“别人可能会问什么”，而是优先处理“哪些误读一旦发生，会大幅降低正确使用率”。因此，B 区必须按理解兑现机制拆分，而非按外部提问表面聚合。

## 3. B 区为何可以在前台与 A 区连续成文

| 维度 | 说明 |
| --- | --- |
| 被拒绝的误解 | B 区若合法，就必须显式自报为“阐释层” |
| 错误原因 | 这会把作者后台分层意识误写成正文元信息 |
| 前台原则 | 模型不需要被告知“这一段属于哪种后台工法”，只需要被正确引导 |
| 后台原则 | 审核与维护仍需保留 A/B 分析区分，以避免职责漂移 |
| 结论 | 前台可连续成文，后台必须可分析分层 |

**证明**：  
A/B 的分层首先是构造与审核意义上的分层，而不是必须对读者显性宣告的版式身份。正文应服务对象本身，而不应陈述作者如何组织对象。故 B 区无需显性写出“本节负责桥接”“以下进入阐释层”等元话语，但在后台证明与审核中，必须保留对其职责的严格区分。

# 第三部分：B 区模块责任矩阵

以下模块，对应前台 B 区正文中真正应当出现的各个分面。  
每个模块都必须通过同一组审查：语义职责、必要性、不可合并性、删除后损失、写作纪律、越界风险。

---

## 4. Operational Priorities

| 维度 | 说明 |
| --- | --- |
| 模块名 | Operational Priorities |
| 语义职责 | 给出 B 区的最小进入顺序，使模型先抓高价值判断框架，而非随机调用局部规则 |
| 必须回答的问题 | 阅读并使用该工具时，什么应先看，什么应后看；哪些内容属于 validity boundary，哪些属于 runtime interpretation boundary |
| 不可替代性 | A 区各节即使都正确存在，也不会自动告诉模型“先抓什么，再抓什么” |
| 不可合并性 | 不能并入 A 区的 Tool Identity 或 Grammar，因为它不是本体定义；也不能并入 Patch Construction Priorities，因为它面向的是全局阅读顺序，而非具体写补丁顺序 |
| 删除后损失 | 模型更容易按旧 diff 经验或局部熟悉项跳读 A 区，导致误读顺序混乱 |
| 写作纪律 | 只给优先级，不重述具体定义；只做顺序锚定，不做后台解释 |
| 越界风险 | 若加入“为什么这样安排”的作者自述，则越界为后台工法说明 |

**结论**：该节必须存在，因为 A 区只定义对象，B 区此处才开始定义“进入顺序”。

---

## 5. High-Risk Misreadings

| 维度 | 说明 |
| --- | --- |
| 模块名 | High-Risk Misreadings |
| 语义职责 | 集中前置拦截那些误读概率高、后果严重、传播半径大且不易自发现的误解 |
| 必须回答的问题 | 哪些理解路径最危险；哪些错误不能一出现就被模型归因为“工具坏了” |
| 不可替代性 | A 区虽给出边界，但不会自动指出“最容易错在哪里” |
| 不可合并性 | 不能并入 Authoring Invariants，因为后者是合法性硬约束，不是误读类型学；不能并入 Failure Surface，因为误读并不等同于运行时失败 |
| 删除后损失 | 模型会把 success summary 误当完整执行日志、把 warning 误当失败、把局部 context mismatch 误当工具失效 |
| 写作纪律 | 使用负面边界句式；优先写高风险误解，不写低价值提醒 |
| 越界风险 | 若把所有可能误会一网打尽，会膨胀为无序的提醒清单 |

**结论**：该节必须存在，因为 B 区首先要拦截的不是“看不懂”，而是“误以为自己已经懂了”。

---

## 6. Patch Construction Priorities

| 维度 | 说明 |
| --- | --- |
| 模块名 | Patch Construction Priorities |
| 语义职责 | 将局部补丁生成时最先需要做出的判断前置组织成稳定序列 |
| 必须回答的问题 | 写补丁时先判断什么；何时需要重读目标内容；如何区分操作类型与局部行构造 |
| 不可替代性 | A 区的 Grammar 与 Invariants 给出合法边界，但不提供构造顺序 |
| 不可合并性 | 不能并入 Operational Priorities，因为一个管全局阅读，一个管局部生成；不能并入 Hunk Discipline，因为前者是任务顺序，后者是 hunk 层纪律 |
| 删除后损失 | 模型更可能在尚未判断操作类型前就开始写局部行，或在内容过期时继续沿用旧上下文 |
| 写作纪律 | 只写构造顺序与前置判断，不重复 grammar 条文 |
| 越界风险 | 若写成教程式操作步骤并引入过多示例，容易从结构纪律滑向培训文案 |

**结论**：该节必须存在，因为“知道什么合法”不等于“知道先怎么构造”。

---

## 7. Hunk Discipline

| 维度 | 说明 |
| --- | --- |
| 模块名 | Hunk Discipline |
| 语义职责 | 将 update hunk 中最容易被旧经验污染的局部认知单独隔离并校正 |
| 必须回答的问题 | hunk body 到底是什么；前缀为何属于语法本体；上下文在 hunk 中承担什么功能 |
| 不可替代性 | Grammar 形式上给出 HunkLine，但不足以阻断模型把 body 当自由文本重述的倾向 |
| 不可合并性 | 不能并入 Authoring Invariants，因为 invariant 是硬规则缓存；此节要处理的是“如何正确理解 hunk”的进入性纪律 |
| 删除后损失 | 模型更易重复 anchor 行、把上下文当改写区、把 header 当正文替代物、把 body 当自由描述区 |
| 写作纪律 | 必须紧贴 hunk 对象；不得扩展为一般 diff 理论；不得夹带作者旁白 |
| 越界风险 | 若写成 parser 级实现猜测，则越界为内部机理臆测 |

**结论**：该节必须存在，因为前一类失败往往首先出现在 hunk 心智模型，而非顶层 patch envelope。

---

## 8. Context Selection

| 维度 | 说明 |
| --- | --- |
| 模块名 | Context Selection |
| 语义职责 | 将“如何选上下文、何时升级上下文、何时停止增加上下文”从一般建议中独立出来 |
| 必须回答的问题 | 默认上下文什么时候够；什么时候 stale、non-unique、over-expanded；何时应改用更强锚点 |
| 不可替代性 | A 区虽有 Anchor Precision Escalation，但仍需 B 区把“选上下文”作为理解动作单独校正 |
| 不可合并性 | 不能并入 Anchor Precision Escalation 的原段，因为前者是本体层升级规则，后者是模型在实际操作中的选择纪律 |
| 删除后损失 | 模型会机械扩上下文、为可读性而扩上下文，或在该升级 header 时继续堆叠无效局部行 |
| 写作纪律 | 只讨论上下文选择；不讨论失败归因总表；不重写 A 区升级规则 |
| 越界风险 | 若加入具体实现细节猜测，则会把“选择纪律”污染为“机理假说” |

**结论**：该节必须存在，因为上下文的错误不是单纯语法错误，而是定位策略错误。

---

## 9. Interpretation Discipline

| 维度 | 说明 |
| --- | --- |
| 模块名 | Interpretation Discipline |
| 语义职责 | 规定模型应如何读取示例、兼容性说明、路径规则与失败诊断，而不做越权推断，也不额外假定二次工件层 |
| 必须回答的问题 | 哪些内容是联合锚点；哪些是范围声明；哪些不能被扩展成更强结论 |
| 不可替代性 | A 区给出事实，但不负责拦截读者对这些事实的过度推论 |
| 不可合并性 | 不能并入 Compatibility Notes，因为兼容性只是其中一类解释对象；不能并入 Success Summary Semantics，因为该节覆盖更广的阅读对象 |
| 删除后损失 | 模型更易把 sample 当定义空间穷尽、把 path rules 当 grammar 扩展、把 failure prose 当无结构抱怨文本 |
| 写作纪律 | 使用“read X as Y, not as Z”句型；保持解释边界，不新增工具语义 |
| 越界风险 | 若试图替工具补全未声明的隐藏机理，则会越界为推测性说明 |

**结论**：该节必须存在，因为文档误读不仅发生在写补丁时，也发生在读文档时。

---

## 10. Failure Re-anchoring

| 维度 | 说明 |
| --- | --- |
| 模块名 | Failure Re-anchoring |
| 语义职责 | 在失败后提供一条优先级明确的重锚路径，防止模型直接跳到工具失效结论 |
| 必须回答的问题 | context mismatch 或局部失败出现后，先检查什么，后检查什么 |
| 不可替代性 | Failure Surface 只定义失败结构，不定义下一步诊断顺序 |
| 不可合并性 | 不能并入 Failure Surface，因为一个是失败对象学，一个是失败后的行动排序；不能并入 Patch Construction Priorities，因为后者是事前构造，前者是事后重锚 |
| 删除后损失 | 模型更易将局部失败统一解释为“工具坏了”，或在未重读内容前反复提交陈旧补丁 |
| 写作纪律 | 必须采用顺序型结构；优先回到 A 区确定项；禁止先行猜测隐蔽实现缺陷 |
| 越界风险 | 若列举过多偶发性底层可能性，则会诱导模型跳过高概率检查项 |

**结论**：该节必须存在，因为高代价错误往往不是“一次失败”，而是“失败后的错误归因”。

---

## 11. Preference Heuristics

| 维度 | 说明 |
| --- | --- |
| 模块名 | Preference Heuristics |
| 语义职责 | 明确那些不属于 legality boundary、但对更稳妥 authoring 有价值的偏好判断 |
| 必须回答的问题 | 在多个合法路径中，哪些选择通常更稳妥 |
| 不可替代性 | 若不独立成节，偏好极易混入 compatibility 或 invariants，导致强弱边界混乱 |
| 不可合并性 | 不能并入 Compatibility Notes，因为兼容性描述的是意图与 runtime 的偏差，不是 authoring 偏好；不能并入 Invariants，因为偏好不是必须条件 |
| 删除后损失 | 模型会丧失一些高价值的稳妥性选择，例如已有路径优先用 Update、rename 优先选 fresh destination、同一区域优先一条强锚 hunk |
| 写作纪律 | 必须使用 `prefer` 语言；不得伪装成 `must`；不得引入未在 A 区支持的强制规则 |
| 越界风险 | 若偏好叠加过多，会伪装成第二套隐性合法性边界 |

**结论**：该节必须存在，因为偏好若不被物理隔离，模型就会把建议误读为硬约束，或把硬约束误读为经验性建议。

---

## 12. Boundary-Preserving Judgments

| 维度 | 说明 |
| --- | --- |
| 模块名 | Boundary-Preserving Judgments |
| 语义职责 | 将若干最关键的“不可越界推断”压缩成最后一层判断纪律 |
| 必须回答的问题 | 从有效补丁到预期结果、从 runtime 偏差到 grammar、从 sample 到合法空间，哪些推断不能做 |
| 不可替代性 | 上述推断若散落在各节，容易失去全局判断力；集中成节可形成最终校准面 |
| 不可合并性 | 不能并入 High-Risk Misreadings，因为后者偏重误解类型，前者偏重判断边界；不能并入 Interpretation Discipline，因为此节承担收束性的最终判断 |
| 删除后损失 | 模型在结尾缺乏一组总括性边界判断，容易在多个局部正确事实之间做错误拼接 |
| 写作纪律 | 必须是短句判断；不展开背景论证；不重复前文定义 |
| 越界风险 | 若发展成哲学式泛论，则会失去工具文本所需的紧度 |

**结论**：该节必须存在，因为 B 区最终需要把模型带回“哪些判断可以成立、哪些判断不能越界成立”的边界线上。

# 第四部分：为何是这一组模块，而不是别的组法

## 13. 为什么不把 B 区写成一个宽泛的 `Patch Writing Guidance`

| 维度 | 说明 |
| --- | --- |
| 原方案风险 | `Guidance` 过宽，容易把全局阅读顺序、局部硬约束、误读拦截、失败回收、偏好判断混成一个大容器 |
| 审核问题 | 一旦内容都挂在 Guidance 名下，就无法判断某一条到底是 legality、heuristic、reading discipline 还是 diagnostic order |
| 当前拆法收益 | 使“优先级”“误读”“构造”“hunk”“上下文”“解释”“失败”“偏好”“判断边界”九类职责物理分离 |
| 结论 | 宽 Guidance 适合作为草稿阶段临时容器，不适合作为最终审核文本的结构单位 |

## 14. 为什么不把 B 区写成案例驱动结构

| 维度 | 说明 |
| --- | --- |
| 原方案风险 | 案例可以帮助进入，但难以覆盖全部高风险误读；案例过强还会反过来替代一般边界 |
| 结构后果 | 文档更像经验手册而非受控阐释层 |
| 审核困难 | 无法判断“这一条是因为某个个案出现过，还是因为它具有高边际价值” |
| 结论 | B 区可以使用少量句法化例示，但不应按案例组织整体结构 |

## 15. 为什么不把 B 区写成“排错手册”

| 维度 | 说明 |
| --- | --- |
| 原方案风险 | 会把 B 区收缩成事后问题处理，而忽略事前进入路径与误读前置拦截 |
| 结构偏差 | 文档整体气质会从“理解兑现层”退化成“故障应对层” |
| 删除损失 | 模型在真正动手前仍缺进入顺序与判断纪律 |
| 结论 | 排错是 B 区的一部分，但不能成为 B 区的总组织原则 |

## 16. 为什么 B 区不讨论接入层、命令包装、传输层

| 维度 | 说明 |
| --- | --- |
| 排除对象 | shell command、MCP、transport、UI 包装、调用渠道 |
| 排除原因一 | 这些属于接入层偶然性，不属于 `apply_patch` 工具语义兑现层 |
| 排除原因二 | 一旦写入 B 区，模型会将工具收缩成某种具体壳子，而非结构化编辑能力 |
| 排除原因三 | B 区的使命是围绕 A 区本体运作，而非补充接入层产品说明 |
| 结论 | 接入层信息若有需要，应独立成文，不能混入 B 区正文或 B 区证明 |

# 第五部分：B 区的排序原则证明

## 17. 为什么 B 区必须排序，而不是全覆盖

| 维度 | 说明 |
| --- | --- |
| 目标问题 | B 区是否应尽量解释更多内容 |
| 结论 | 不应追求全覆盖，应追求高边际价值阐释 |
| 原因一 | 模型注意力与上下文资源有限 |
| 原因二 | 说明越多不必然越好，低价值说明会稀释高价值边界 |
| 原因三 | 真正高代价的误读点数量有限，且可被前置处理 |
| 审核含义 | B 区新增内容时，必须首先回答“它是否属于高边际价值阐释” |

## 18. B 区内容进入的五维判据

一条内容若要进入 B 区，至少应通过以下五维评估：

| 维度 | 问题 | 审核阈值 |
| --- | --- | --- |
| 误读概率 | 模型首次接收时是否高概率在此处跳步或套旧框架 | 高概率项优先 |
| 后果严重度 | 一旦误解，是否会显著降低正确使用率 | 高严重度项优先 |
| 传播半径 | 该误解是否会污染后续多个判断与操作 | 高半径项优先 |
| 隐蔽性 | 该误解是否不易在后续自行暴露 | 高隐蔽项优先 |
| 阐释成本 | 讲清此点是否成本适中且收益显著 | 高回报项优先 |

**结论**：B 区不是“凡可说皆说”的空间，而是“有限资源下优先讲什么”的排序空间。

# 第六部分：B 区的越界判据

## 19. 四类典型越界

### 19.1 新增本体承诺
若某段 B 区文字引入了 A 区未给出的必要特征、运行承诺或解释边界，则越界为概念篡改。

### 19.2 例子偷换定义
若某个高频场景被默认为概念本身，而不再明确它只是示例，则 B 区越界为特例绑架。

### 19.3 类比替代边界
若某个类比或经验说法使模型忘记回到 A 区条文，则 B 区越界为高迷惑性支架。

### 19.4 为了顺滑删掉关键约束
若为了让首读更自然，而删去限制条件、适用域或排除条件，则 B 区越界为删减式失真。

## 20. B 区的三项质量标准

| 维度 | 含义 |
| --- | --- |
| 忠实性 | 是否始终保持对 A 区的可回收性与边界忠实 |
| 桥接性 | 是否真正降低目标模型的误读率与进入门槛 |
| 收束性 | 是否能在必要处停下，而不无限膨胀解释 |

任何仅仅“读起来更顺”的改写，若不能同时通过这三项标准，都不应视为高质量 B 区。

# 第七部分：B 区完整写作纪律总表

## 21. 总纪律

| 纪律 | 内容 |
| --- | --- |
| 忠实纪律 | 不新增 A 区未声明的概念承诺、运行承诺或解释边界 |
| 排序纪律 | 优先处理高误读概率、高严重度、高半径、高隐蔽性、低成本高回报的内容 |
| 分面纪律 | 一个标题只承载一个主要语义职责 |
| 边界纪律 | legality、heuristic、misreading、reading discipline、diagnostic order 不得混写 |
| 句子纪律 | 一句只承担一种主要动作：前置、拦截、排序、重锚、判断 |
| 元信息纪律 | 不写作者自我动作；不写维护方法；不写“本节负责什么”的后台说明 |
| 示例纪律 | 示例若出现，只能做联合锚点，不得承担定义职责 |
| 术语纪律 | 与 A 区接口的术语保持稳定，不做修辞性同义替换 |
| 回收纪律 | 每个局部桥接都应能回指 A 区，而非停留在局部经验上 |
| 排除纪律 | 不写接入层偶然性，不写 transport，不写产品包装，不写后台工法解释 |

# 第八部分：删除测试与合并测试

## 22. 删除测试

审核 B 区时，建议逐节问下列问题：

1. 删除这一节后，损失的是“概念边界”，还是“理解兑现能力”？
2. 若仅损失概念边界，则该内容应当回到 A 区，而不属于 B 区；
3. 若既不损失边界，也不损失理解兑现，则该内容应当被删除；
4. 若删除后模型更易误解 A 区、更易错误归因失败、更易把局部经验当一般规则，则该内容应保留在 B 区。

## 23. 合并测试

建议逐对检查下列模块是否可以合并：

- `Operational Priorities` 与 `Patch Construction Priorities`
- `High-Risk Misreadings` 与 `Boundary-Preserving Judgments`
- `Hunk Discipline` 与 `Context Selection`
- `Interpretation Discipline` 与 `Failure Re-anchoring`
- `Preference Heuristics` 与 `Authoring Invariants`

**判断原则**：  
若合并后会使“硬约束 / 偏好 / 误读 / 解释 / 失败后顺序”中任意两类职责重新混写，则不得合并。

# 第九部分：最终审核立场

本文档主张的，不是“再写一篇更长的说明”，而是建立一份与 A 区证明文档平行的 B 区构造证明。  
其核心结论如下：

1. A 区负责概念保真，B 区负责理解兑现；
2. A 区的完备性并不自动消除 B 区的必要性；
3. B 区不能写成第二份 A 区，也不能写成 FAQ 或排错仓库；
4. B 区必须按高边际价值阐释原则排序；
5. B 区必须始终围绕 A 区运作，而不得改写 A 区；
6. B 区前台可以与 A 区连续成文，后台则必须保留可审核的分层证明；
7. B 区各模块之存在，均可通过必要性、不可合并性、删除后损失与越界测试加以验证。

# 第十部分：供团队审核的终版问题清单

以下问题可作为团队最终审稿时的统一清单：

1. B 区每一节是否都对应一个明确且不可替代的理解兑现职责？
2. 是否存在任何一节，只是在重复 A 区，而没有新增理解兑现价值？
3. 是否存在任何一节，实际在偷偷新增本体承诺或运行承诺？
4. 是否把硬约束、偏好、误读、解释、失败顺序混写到了同一节？
5. 是否优先处理了高代价误读，而不是把说明资源浪费在低价值补充上？
6. 是否存在作者后台元话语泄露到正文中的情况？
7. 是否存在接入层、产品包装层、传输层偶然性混入 B 区的情况？
8. B 区是否能在不改写 A 区的前提下，显著提高模型首次正确使用工具的概率？
9. 若删除任一模块，其损失是否可以被明确说明，而不是仅凭“感觉上更完整”？
10. 若以上各项均可通过，则 B 区可视为具有独立而稳定的审核地位。
